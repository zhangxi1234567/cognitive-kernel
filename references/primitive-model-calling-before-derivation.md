# Primitive: Model Calling Before Derivation

## Pattern

A hard-looking problem becomes manageable once it is routed into a known model, archetype, or canonical picture before raw derivation begins.

## Trigger Signal

- the notes or solver naturally want to say "这是个某某模型"
- the surface form looks novel but the relation web feels familiar
- the full derivation is long while one named picture or setup would organize the whole problem
- a standard geometric, functional, combinatorial, or physical model seems to carry the decisive structure
- once the model is called, many later steps look like routine readout rather than invention

## Level-Agnostic Core

Do not derive from raw chaos if a correct model already compresses the relation.
Call the model first.

## Control Mechanism

The move works because:

- the model is a compressed carrier of the controlling relation
- the model kills novelty by routing the problem into a known structural family
- once the right model is called, the remaining work is usually readout, sealing, or light adaptation

## Preserved Structure

Typical preserved items:

- the load-bearing relation shared by the raw problem and the model
- admissible geometry / behavior class
- target-relevant map from the raw problem to the model frame
- constraints that survive after the model call

## Earliest Honest Layer

Before deriving, ask:

- what is this secretly a model of?
- if I had already seen the right picture, what family would it belong to?
- which known setup would kill the most novelty immediately?

## Fast Move

1. Identify the likely model, archetype, or canonical picture.
2. Map the raw problem into that model.
3. Read the key relation inside the model first.
4. Only then derive the residue needed to match the original target.

## Compression Move

Typical burden deletion:

- novel shell -> known model
- raw derivation -> model readout
- many local relations -> one organized template
- ad hoc setup -> canonical picture with built-in structure

## Level Transfer

- low-complexity form: call the familiar picture or standard geometric setup
- compact formal form: route to a named model family or archetype
- advanced formal form: effective model, canonical representation, or reduced theory before full derivation

## Cross-Domain Analogue

- debugging: route a messy failure into a known bug archetype before inspecting every trace
- strategy: classify a messy situation into a known scenario model before optimizing details
- science: choose the effective model or regime picture before full formal treatment

## Where It Transfers

- geometry via standard triangles, flagpole models, cross-plane models, reflection models
- function problems via mother functions and symmetry models
- combinatorics via standard occupancy / matching / exact-cover models
- space geometry via section, complement, or canonical-solid models
- any domain where a correct model call deletes more work than raw derivation

## Failure Mode

- the chosen model is only cosmetically similar
- important constraints are lost in the mapping
- the model suggests a route but the target needs a different control relation
- the solver hides behind a model name without stating what it preserves

## Verification Bill

State:

- what exact model is being called
- what raw constraints survive in the model
- why the model carries the target-relevant structure
- what still has to be translated back
