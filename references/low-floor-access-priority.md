# Low-Floor Access Priority

This file defines a weak package preference:

- if the same controller can still be truthfully carried at a lower honest layer, the package should not reward staying too early in a heavier formal shell

Its purpose is:

- keep the route accessible to smaller carriers
- allow more primitive combinations to become naturally visible
- without prescribing any descent order

## Core Claim

Some routes are not wrong,
but they are still heavier than necessary.

If a lower honest layer can still carry the same controller,
the package should preserve weak pressure toward that layer.

This is not:

- anti-rigor

It is:

- anti-unnecessary heaviness

## What This Can Reveal

Lower honest layers often make more candidate skills visible:

- symmetry
- special value
- boundary
- projection
- comparison
- counting

So low-floor access matters not only for explanation,
but also for natural skill coactivation.

## What The Package May Preserve

The package may preserve:

- `low_floor_access_priority`
- `bottom_layer_priority`
- `composite_shell_risk`

These are:

- weak preferences

Not:

- forced simplification orders

## What Must Not Happen

The package must not prescribe:

- “always go to the elementary version first”
- “always convert to a toy model”
- “always ask the child-level question”

Those would become scripts.

## Relation To Existing Docs

Use:

- `lowest-honest-layer.md`
- `knowledge-ingress.md`
- `minimal-executable-core.md`

This file lifts their shared package-level implication:

- lower honest access can improve both clarity and route competition

## Final Rule

The package may prefer lower honest access when it keeps the true controller alive.

It must not force a descent ritual.
