# Compat-Active Anti-Regrowth

This file defines the compatibility-active form of anti-regrowth pressure.

Its purpose is:

- make compat-only execution less likely to grow back into ordinary solution flow after a real reduction
- preserve reduced-route integrity without visible policing

## Core Claim

Compat-only execution should already resist:

- chapter-default rebuilds
- full-object reconstruction
- derivation tails that erase the earlier reduction

after a good carrier has already appeared.

## What This Means In Compatibility Form

At the compatibility-active level, anti-regrowth means:

- a route is cooled when new burden grows faster than old burden dies
- route continuation is biased toward staying on the small carrier
- settlement is not trusted if the main body has drifted back to ordinary flow

This is not:

- a visible anti-regrowth checklist
- a forced interruption

## Allowed Compatibility Pressure

The compatibility layer may already preserve:

- `remaining_fake_burden`
- `carrier_integrity_pressure`
- `target_complete_readiness`
- `competition_saturation`

These are enough to support:

- weak live anti-regrowth pressure

without:

- a visible regrowth audit

## What Must Stay Forbidden

Do not turn anti-regrowth into:

- “check if this is textbook now”
- a required burden-audit sequence
- a scripted return-to-small-carrier command

## Final Rule

Compat-only execution should already resist slipping back into ordinary flow.

It must not do so by visible policing.
