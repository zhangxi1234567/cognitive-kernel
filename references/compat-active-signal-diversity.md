# Compat-Active Signal Diversity

This file defines the compatibility-active form of signal diversity.

Its purpose is:

- let one problem shell register more than one lawful trigger signal at once
- stop early flattening of the object into one single interpretation

## Core Claim

Compat-only execution should already allow:

- one object to emit several compatible route signals
- those signals to remain locally distinguishable
- route choice to benefit from that plurality before consolidation

This is not:

- endless reinterpretation

It is:

- weak trigger plurality

## What This Means In Compatibility Form

At the compatibility-active level, signal diversity means:

- one structure may simultaneously look like local readout, limit pressure, and direct-readout opportunity
- one problem need not collapse to the first named trigger
- local candidate differences stay informative slightly longer

## Allowed Compatibility Pressure

The compatibility layer may already preserve:

- `signal_diversity`
- `informative_divergence`
- `candidate_field_liveness`

These are enough to support:

- weak signal plurality

without:

- visible multi-trigger scripts

Interpretation rule:

- signal diversity means more than one lawful trigger may remain locally visible
- not that the host should enumerate or orchestrate those triggers

## Final Rule

Compat-only execution should already allow more than one lawful trigger signal to stay locally real.

It must not expose that plurality as workflow.
