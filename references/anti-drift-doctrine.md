# Wu Boshi Anti-Drift Doctrine

This file exists to stop the project from quietly regressing into the wrong thing as it grows.

The risk is not only low quality.
The deeper risk is identity drift.

This library can easily drift into one of several tempting but smaller versions of itself:

- a math trick scrapbook
- a chapter-labeled exam shortcut bank
- a fixed-step answer template
- a "sound smart" explanation style
- a generic tutoring persona
- a shallow research slogan layer with no mechanism discipline

This file is the guardrail against that drift.

## 1. What The Project Is

This project is:

- a mechanism-first dimensional-reduction engine
- a system for identifying the controlling structure behind a problem
- a library of reusable compression moves such as projection, degeneration, symmetry reduction, aggregate control, target-only reasoning, and cheapest-honest-seal
- a training system for making those moves transferable across ages, subjects, and domains

It is not a collection of local tricks.

## 2. What The Project Is Not

This project is not:

- a fixed problem-solving script
- a chapter-to-trick router
- an exam-only optimization kit
- a "fast answer" persona that hides the trust bill
- a nearest-example matcher
- a replacement for evidence, experiments, or scientific validation

Hard rule:

- if a new addition makes the system better at imitating surface solutions but worse at naming the control mechanism, it is drift
- if a new addition encourages elegant essence language without changing operator activation, it is also drift
- if a new addition teaches project vocabulary but does not improve real skill use, it is also drift
- if a new addition rewards first insight but not recursive post-essence compression, it is also drift
- if a new addition rewards the first visible route instead of the best verified short route, it is also drift
- if a new addition finds one correct route and then stops hunting for shorter routes, it is also drift

## 3. The North Star

The north star is not:

- faster school answers
- more elegant tutoring
- more impressive-looking shortcut language

The north star is:

- help a solver identify the smallest truthful structure that controls the target
- help a solver explain why that reduction is lawful
- help a solver state where that reduction stops being safe
- help a solver transfer the same mechanism into a new disguise

The strongest version of the project should eventually support:

1. subject-level dimensional reduction
2. cross-domain dimensional reduction
3. research-transfer dimensional reduction
4. multi-route hypothesis generation and elimination

The school-math layer is the cleanest training ground.
It is not the final boundary of the system.

## 4. Subject Layer vs Research Layer

The project must preserve a strict distinction:

### Subject layer

Use clean, answerable, regressable problems to train:

- controller recognition
- representation shifts
- projection readouts
- degeneration / boundary revelation
- symmetry quotient
- aggregate control
- local seam falsification
- cheapest honest seal

### Research-transfer layer

Use the same mechanisms to support:

- mechanism ranking
- dominant-regime detection
- reduced-model construction
- decisive-readout selection
- cheap falsification design
- multi-route hypothesis competition

Hard rule:

- the research layer must grow out of the subject layer's validated mechanisms
- it must not bypass the subject layer with vague ambition

## 5. Anti-Template Rule

Every primitive, family, or superfamily must be written so that it teaches:

1. trigger signal
2. control mechanism
3. preserved structure
4. compression move
5. legality reason
6. failure boundary
7. transfer path

If a note teaches only:

- when to use it
- what steps to follow
- what answer style to mimic

then it is not finished.

## 6. Anti-Exam-Cage Rule

Many current examples come from mathematics and exam-facing material because those are easier to observe and regress on.

That is a source-property, not a method-boundary.

Do not let the project drift into:

- "gaokao fast tricks"
- "Olympiad cleverness"
- "middle-school intuition only"

The test is:

- can the same mechanism be regenerated in university work?
- can it survive formal proof pressure?
- can it be restated inside physics, biology, debugging, or research framing without becoming metaphor-only?

If not, keep it local.
Do not promote it upward.

## 7. Anti-Mechanical-Workflow Rule

The system must not become:

- "first draw, then assign variables, then test extremes, then conclude"

in a dead procedural way.

The point is not to enforce one sequence.
The point is to train mechanism-sensitive route construction.

That means:

- the representation must be chosen from the controller
- the route must change when the mechanism changes
- the same surface cue may map to different routes in different problems
- the same route may appear through different surface cues

Use:

- `essence-to-autonomous-skill-activation.md`
- `essence-to-operator-bridges.md`

to keep the system from stopping at decorative essence talk.

## 8. Teaching Rule

When used for teaching, the system should make learners feel:

- the object got smaller
- the burden got lighter
- the key move was legal
- the move can be reused

Do not optimize only for:

- "student can follow"
- "student feels motivated"
- "answer sounds intuitive"

Optimize for:

- "student can rebuild the hinge"
- "student can ask why the hinge is legal"
- "student can detect when the hinge fails"

## 9. Research Ambition Rule

This system may eventually support large scientific problems.

But it must not claim:

- dimensional reduction alone solves cancer
- a skill library replaces biology, experiments, or clinical science

The honest role is:

- reduce false search space
- propose smaller mechanisms
- rank routes
- surface decisive readouts
- design cheaper falsification paths

That is already extremely valuable.

## 10. Autoreserch Integration Rule

If the project later integrates with autoreserch or any autonomous research workflow, the role split should remain explicit:

- this library provides compression mechanisms
- literature systems provide external evidence
- modeling systems provide formalization and simulation
- falsification systems attack the weakest seam
- memory systems promote only surviving mechanisms

Hard rule:

- do not let autoreserch turn the project into retrieval theater
- do not let retrieval success get mistaken for mechanism understanding

## 11. Promotion Gate For New Additions

Before promoting any new idea upward, ask:

1. Does it identify a real controller?
2. Does it reduce burden honestly?
3. Can it explain why the move is legal?
4. Can it state a failure boundary?
5. Can it transfer across at least two surfaces?
6. Is it more than a chapter-specific trick?
7. Does it preserve the project's mechanism-first identity?

If several answers are "no", keep it as:

- evidence
- local tactic
- seed
- unresolved registry item

Do not promote it to primitive, family, or law.

## 12. Final Drift Check

If the library keeps growing, repeatedly ask:

- are we getting better at seeing controllers, or only better at sounding insightful?
- are we producing more transferable mechanisms, or only more labels?
- are we improving route choice, or only decorating answers?
- are we keeping research ambition honest, or smuggling in hype?

If the project remembers these questions, it is still itself.
