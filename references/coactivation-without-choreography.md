# Concurrent Weak Presence Without Choreography

This file defines a compatibility rule:

- several locally relevant possibilities may remain weakly present together without becoming a visible script

Its purpose is:

- preserve local plurality without forcing enumeration
- preserve natural cooling without forced pruning rituals

## Core Claim

Concurrent weak presence means only this:

- more than one locally relevant possibility has not yet cooled away

It does not mean:

- compare everything
- preserve everything
- narrate everything
- keep a visible candidate board

## What May Be Preserved

The package may preserve:

- that local plurality has not yet cooled
- that one visible route may still be too thick
- that natural consolidation has not yet happened

The package may not preserve:

- candidate tables
- explicit survival criteria
- explicit kill criteria
- explicit co-presence scoring

## Final Rule

Concurrent weak presence is a local runtime phenomenon.

It must not become an explicit object-management procedure.
