# Wu Boshi Dimensional Reduction Instinct

This file defines a deeper project target than:

- explicit workflow
- explicit prompting
- explicit route control

Its purpose is to make one thing explicit:

- the project does not merely want the model to know reduction moves
- it wants the model to increasingly *want* smaller truthful structure

## Core Claim

The project should not be satisfied with a model that can:

- name symmetry
- mention boundary
- say “the essence is...”
- perform one local reduction when told

The deeper target is:

- the model increasingly favors smaller truthful structure
- the model increasingly avoids settling too early for still-thick carriers
- the model continues compressing only while further reduction remains honest

Short version:

- not reduction as script
- but reduction as instinct

## What This Instinct Is

Dimensional-reduction instinct is the tendency to:

- distrust decorative surface
- prefer smaller carriers
- resist premature settling on intermediate structure
- keep open the possibility of a deeper controller after the first good insight
- stop only when the remaining structure is truly load-bearing

It is not:

- mechanical repetition of “can I reduce further?”
- a visible checklist
- forced recursion imposed from outside

It is:

- a stable internal preference
- a compression appetite
- a refusal to settle too early for middle-layer structure

## What This Instinct Is Not

It is not:

- endless simplification theater
- mandatory repeated questioning
- ritualized self-interrogation
- blind pressure to reduce even when the route is already honest

The project does not want:

- infinite regression
- compression for its own sake

The project wants:

- the model to retain pressure toward smaller truthful structure
- and to keep compressing only while truth-preserving gains remain available

## Why This Matters

Without this instinct, the system may:

- catch one real insight
- improve the route somewhat
- then stop too early

This creates a common failure mode:

- local intelligence
- incomplete compression

The route looks better than ordinary solving, but it still has not reached:

- the smallest controller
- the strongest main carrier
- the cleanest readout

This is not the mature target.

The mature target is:

- first insight does not automatically end the search for a smaller truthful carrier
- continuous reduction remains possible while it is still honest
- final settling happens only after the route has become genuinely small enough

## The Real Unit Of Growth

The project should not only grow:

- more skill files
- more cards
- more route examples

It should grow:

- stronger intolerance for fake complexity
- stronger attraction to lower honest layers
- stronger sensitivity to shared controllers
- stronger habit of continuing compression after the first successful jump

This means the deepest unit of growth is not:

- skill inventory

It is:

- cognitive taste

## Reduction Taste

The project should cultivate preference for:

- one controller instead of many moving parts
- one readout instead of full reconstruction
- one seam instead of everywhere-proof
- one main carrier instead of route sprawl
- one minimal seal instead of decorative derivation

And resistance to:

- carrying full objects without need
- solving every variable
- stopping at a middle shell
- repeating standard flow after one good idea
- treating local success as final compression

## How This Differs From Workflow

Workflow depends on:

- external control

Instinct depends on:

- internal preference

The project wants the second.

## Failure Modes

### Failure A: One-Layer Reduction

Shape:

- one real reduction happens
- then the route stops

Problem:

- the model saw structure
- but did not continue inward

### Failure B: Decorative Compression Language

Shape:

- essence language appears
- route appetite does not change

Problem:

- rhetoric replaced instinct

### Failure C: Externalized Recursion

Shape:

- the system only keeps reducing when explicitly instructed

Problem:

- workflow replaced internal drive

### Failure D: Over-Compression Addiction

Shape:

- the model keeps reducing after the load-bearing structure is already reached

Problem:

- compression became vanity instead of truth-preserving simplification

## Success Condition

The instinct is working when route behavior increasingly becomes smaller without needing externally imposed recursion rituals.

That is different from:

- merely obeying a reduction recipe

## Final Rule

The project should increasingly judge itself by this question:

- does the model merely know reduction moves,
- or has it developed an internal need to keep searching for the smallest truthful structure?

If the answer is only the first, the project is still shallow.

If the answer becomes the second, the project is becoming what it actually wants to be.
