# Resonance Layer

This file defines the project's positive account of resonance.

Its purpose is:

- describe how several nearby mechanisms may become locally visible together
- preserve that visibility without turning it into a composition script
- distinguish informative resonance from decorative resonance

## Core Claim

Resonance is not a plan to use several techniques together.

Resonance is:

- a local increase in mutual visibility among nearby lawful mechanisms
- caused by one live shell still carrying more than one kind of surviving burden

It is not:

- a required multi-skill combination
- a proof of depth by plurality
- a reason to narrate several moves performatively

## Mechanism Neighborhood

A mechanism neighborhood is a small local field of nearby lawful possibilities.

Examples may include:

- boundary
- limit
- symmetry
- special value
- direct comparison
- projection
- direct readout
- normalization

The point is not that these must be used.

The point is that one live carrier may make several of them locally visible.

## Informative Resonance

Resonance is informative when:

- different nearby mechanisms illuminate non-identical surviving burden
- the local field is still helping distinguish what the shell really is
- the cluster is making a smaller truthful carrier easier to notice

Short version:

- informative resonance sharpens the route

## Decorative Resonance

Resonance is decorative when:

- plurality no longer deletes new burden
- several nearby mechanisms are being kept alive only to look rich or deep
- the local field has stopped changing the carrier choice

Short version:

- decorative resonance should cool

## Cooling Rule

Resonance does not need to become an explicit combination.

It may:

- cool away completely
- collapse into one thinner carrier
- leave behind only one decisive readout

That cooling should happen naturally.

It must not be managed as:

- a checklist
- a stage gate
- a composition requirement

## Resonance-To-Carrier Collapse

Healthy resonance often ends when:

- one carrier absorbs the useful burden deletion of the local cluster
- the remaining nearby mechanisms become redundant
- the route becomes smaller without losing honesty

This is not:

- the package selecting a winner by script

It is:

- the field becoming locally thinner

## Host Constraint

The host may preserve:

- whether local resonance still appears informative
- whether nearby mechanisms still illuminate distinct burden
- whether the cluster has already cooled into one carrier

The host must not preserve:

- a reusable resonance recipe
- a required mechanism bundle
- an ordered cluster-unfolding script

## Final Rule

The project should allow local resonance to appear, intensify, cool, or disappear.

It must not convert resonance into a reusable skill-combination template.
