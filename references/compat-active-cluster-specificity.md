# Compat-Active Cluster Specificity

This file defines the compatibility-active form of cluster specificity.

Its purpose is:

- keep lower-level clusters from blurring into one generic “many skills alive” fog
- preserve local distinctness without forcing explicit labeling

## Core Claim

Compat-only execution should not only preserve:

- that something else may still be alive

It should increasingly preserve:

- what kind of nearby cluster is alive

Examples:

- boundary-limit flavored
- special-value-symmetry flavored
- balance-normalization flavored

This remains:

- weak
- local
- provisional

## What This Means In Compatibility Form

At the compatibility-active level, cluster specificity means:

- different nearby constellations should not collapse too early into one generic candidate cloud
- the system can feel whether the local neighborhood is more seam-like, more symmetry-like, or more balance-like

This does not mean:

- the model must name the cluster
- the model must enumerate cluster types

## Allowed Compatibility Pressure

The compatibility layer may already preserve:

- `lower_level_cluster_specificity`
- `fundamental_microcluster_visibility`
- `signal_diversity`

These are enough to support:

- weak local cluster distinctness

without:

- visible taxonomy replay

## Final Rule

Compat-only execution should increasingly preserve distinct local cluster flavor.

It must not expose that flavor as a forced label.
