# Non-Workflow Compatibility Contract

This file defines a hard compatibility rule for exposing this project to:

- Codex
- Claude Code
- other agent runtimes

The rule is:

- compatibility must not turn the project into external workflow control

## Core Claim

This project must not be exposed to an agent as:

- a mandatory sequence of steps
- a fixed recursive script
- a topic-triggered skill router
- a human-controlled reduction checklist

It must be exposed as:

- a field of candidate attack angles
- a bias toward smaller truthful structure
- a runtime that allows weak local presence and strong competition
- a deeper instinct for continued honest compression
- a field that does not pre-close whether one primitive or several primitives remain locally relevant
- a field that does not pre-close whether the current layer is shell or lowest honest layer

- internalized mechanism search

## What The Compatibility Layer May Do

The compatibility layer may:

- bias attention toward structural tension
- remind the agent to distrust decorative shell
- encourage candidate operator competition
- discourage regrowth into ordinary flow
- preserve evaluation standards

The compatibility layer may not:

- dictate a visible solve order
- demand explicit self-narration of every internal step
- require repeated recursion as a ritual
- pre-assign a named-technique composition plan
- pre-assign a shell-stripping ladder

## Weak Guidance, Strong Selection

The correct compatibility posture is:

- weak guidance
- strong selection

That means:

- the agent is lightly biased toward controller-first thinking
- several candidate reductions may become active
- the best route must win by actual burden deletion and honest verification

Not by:

- prompt obedience
- fixed step count
- declared alignment with project language

## Internal Drive Over External Prompting

The project should increasingly rely on:

- internal dissatisfaction with still-thick structure
- internal attraction toward smaller main carriers
- internal sense that a route has not yet reached the true controller

It should increasingly avoid reliance on:

- external recursion instructions
- visible prompts to ‚Äúgo deeper‚Ä?- human-imposed reduction loops
- human-authored decisions about single-primitive versus multi-primitive composition
- human-authored decisions about when stripping has bottomed out

## Compatibility Red Flags

The project is being exposed incorrectly if the compatibility layer causes:

- repeated explicit ‚Äúreduce further‚Ä?commands
- a visible numbered reduction pipeline
- skill invocation by topic keyword
- all tasks to start with the same named moves
- users to see project steps instead of project behavior

## Compatible Runtime Shape

A good compatible run often preserves these behavioral properties:

- structural tension is noticed
- several attack angles become weakly active
- candidate routes compete
- weak routes die quickly
- one route takes the lead
- that route continues compressing as long as truth-preserving gains remain
- the route stops when the remaining structure is honestly load-bearing

The key point:

- these are the shape of internal behavior
- not a script to be externally imposed

The compatibility layer must not close off:

- stay with one primitive when one primitive is enough
- several primitives remaining weakly present together when one primitive does not dissolve the shell
- continue stripping when the current carrier still feels compositely loaded
- stop stripping when further descent would cease to be honest
