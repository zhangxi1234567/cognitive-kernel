# Wu Boshi Cross-Domain Dimensional Reduction

This file defines how the project should move from:

- mathematics as the cleanest training ground

to:

- dimensional reduction across many domains

The project should not be understood as:

- a math-only compression library

It should be understood as:

- a cross-domain mechanism library whose cleanest early evidence happens to come from mathematics

## 1. Core Principle

The project should not transfer by copying math shells into every new domain.

It should transfer by regenerating the same deep controller in a new native surface.

That means:

- same compression logic
- different visible object
- different domain-native representation
- different domain-native verification

Short version:

- same mechanism
- new surface

## 2. Mathematics Is The Training Ground, Not The Boundary

Mathematics remains the best early training surface because:

- mechanisms are clean
- preserved structure is visible
- verification is cheap
- route quality is easier to compare

But that does not mean:

- the project ends in mathematics

The real ambition is:

- any domain where complex surface burden can be compressed to a smaller truthful controller

## 3. What Should Transfer

The project should increasingly transfer things like:

- controller selection
- fake-degree elimination
- boundary revelation
- target-only reduction
- projection readout
- aggregate control
- relation before object
- local seam controls global
- cheapest honest seal
- main-carrier choice
- continuous dimensional-reduction chaining

These are not math tricks.

They are cross-domain cognitive operators.

## 4. What Must Not Transfer Naively

The project must not simply export:

- school-math wording
- contest-style labels
- symbolic solve rituals
- chapter-shaped templates

That would create fake transfer.

Real transfer requires:

- new object language
- new readout language
- new trust bill

while preserving the same underlying controller.

## 5. Cross-Domain Examples

### Programming

Math-style burden:

- many variables
- many equations
- many branches

Programming regeneration:

- many modules -> one critical state transition
- many logs -> one first divergence point
- many functions -> one controlling contract
- many symptoms -> one smallest reproducer

Programming also adds a recurring extra burden:

- object-class mixing

Examples:

- hostname versus IP
- trusted versus untrusted input
- runtime state versus config state

Use:

- `object-class-boundary-check.md`
- `exception-first-seam-check.md`

to keep a good carrier from overgeneralizing across incompatible classes.

The same deep operators still apply:

- collapse to one carrier
- isolate the first failure seam
- keep only the target-relevant state relation

### Debugging

Math-style seam:

- first dangerous value
- boundary
- contradiction point

Debugging regeneration:

- first failing input
- first invariant break
- first divergence in expected versus actual state

The same deep move survives:

- local seam controls global

### Research

Math-style target:

- one value
- one relation
- one inequality

Research regeneration:

- one decisive readout
- one dominant mechanism
- one falsifying experiment
- one reduced subsystem

The same deep operators survive:

- projection to decisive observable
- dominant mechanism readout
- target-only experiment design
- cheapest honest falsification

### Teaching

Math-style low floor:

- lower honest layer

Teaching regeneration:

- express the same controller in age-appropriate language
- keep the hinge executable, not only explainable

The same deep rule survives:

- lowest honest layer, not lowest cosmetic layer

## 6. The Right Runtime Meaning

Cross-domain transfer should not become:

- a domain routing checklist

It should remain:

- a flexible regeneration process

The model should increasingly ask:

- what is the object here?
- what is the target here?
- what is the same controller wearing new clothes?
- what is the smallest carrier in this domain?
- what is the domain-native cheap seal?
- what object-class split or exception seam could still break this elegant route?

These are orientation questions, not fixed visible steps.

## 7. Final Rule

The project succeeds cross-domain when:

- the same reduction logic survives
- the surface representation changes honestly
- the trust bill changes honestly
- the thinking stays free rather than becoming a copied subject template
