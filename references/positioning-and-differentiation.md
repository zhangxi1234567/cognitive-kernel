# Wu Boshi Positioning And Differentiation

This file states what kind of project this is, what kind of project it is not, and how it differs from nearby open-source ecosystems.

It exists because this library can easily be misunderstood.

It can be mistaken for:

- a math shortcut collection
- a tutoring style pack
- a generic critical-thinking prompt
- a research-agent wrapper
- a skills marketplace entry with a distinctive tone

It is not those things.

## 1. Core Positioning

This project is a:

- cognitive skill library
- mechanism-first reasoning system
- dimensional-reduction engine
- cross-domain primitive library

Its purpose is not to teach a model how to replay fixed steps.

Its purpose is to teach a model how to:

- identify the controlling structure behind a problem
- compress the problem lawfully into a smaller truthful object
- explain why that compression is legal
- state where that compression stops being safe
- regenerate the same mechanism under a new surface form

Short version:

- not `How to do`
- but `How to think`
- and also `Why this move is lawful`

Deeper version:

- not a solving workflow
- but an exploration substrate
- not just answer production
- but mechanism-seeking cognition

## 2. The Main Difference From Typical Skill Libraries

Most current AI skill ecosystems teach models:

- how to execute actions
- how to follow workflows
- how to call tools
- how to produce better-formatted answers

This project teaches something deeper:

- how to search for the hidden controller
- how to choose a better representation
- how to reduce burden before solving
- how to name the legality basis of a shortcut
- how to transfer a mechanism across domains and rigor levels

In that sense:

- most tools teach `how to do`
- this library tries to teach `how to think`

But even that is still incomplete.

The deeper distinction is:

- many frameworks teach "think more"
- this library tries to teach "what kind of mechanism to look for"

## 3. Not A Fixed SOP

This project must not become a dead checklist such as:

- first draw
- then assign values
- then test extremes
- then conclude

That would miss the point.

The point is not one fixed workflow.
The point is flexible route construction from reusable primitives.

Even "route construction" is still only the visible surface.

The deeper aim is to cultivate cognitive operators that change how the model sees a problem before any route is chosen.

Examples of such primitives include:

- projection readout
- degeneration / boundary revelation
- symmetry quotient
- aggregate controls target
- relation before object
- target-only reduction
- local seam controls global
- dominant mechanism readout
- cheapest honest seal

These are not "study habits."
They are operational compression moves.

The project now also makes explicit that:

- catching the essence is not enough by itself
- the recognized essence should activate candidate operators and skills
- one problem may trigger one move or several cooperating moves

Use:

- `essence-to-autonomous-skill-activation.md`
- `essence-to-operator-bridges.md`

to keep that distinction explicit.

## 4. Cross-Domain Isomorphism

One of the defining ambitions of the project is cross-domain isomorphism.

That means:

- the same deep mechanism should survive across multiple surfaces
- the same primitive should be recognizable in elementary, advanced, and research-flavored settings

Examples:

- derivative is not only a calculus formula
- it can also be seen as local motion, velocity, acceleration, or rate-of-change reasoning
- area is not only a memorized formula
- it can be read as a lower-dimensional control relation
- symmetry is not only a contest trick
- it can become invariance, relabeling, quotienting, canonical framing, or conserved structure

The goal is not analogy theater.
The goal is structure-preserving transfer.

Hard rule:

- if a claimed transfer only sounds poetic but does not preserve the controlling mechanism, it is not real transfer

Use:

- `cross-domain-dimensional-reduction.md`

to keep cross-domain ambition tied to true mechanism regeneration rather than slogan-level analogy.

## 5. Not A Trick Index

This library must never be reduced to:

- "triangle area uses this trick"
- "conic problems use that trick"
- "derivative questions use these five steps"

It should instead teach:

- what signal suggests a hidden controller
- what operation reveals that controller
- what is preserved under the reduction
- why the reduction is lawful
- when the route breaks
- where else the same primitive should reactivate

This is why the unit of growth should be:

- primitive
- family
- superfamily
- law
- mechanism

not:

- chapter
- trick name
- exam label

## 6. Why This Can Help Weaker Models

This library does not magically upgrade a weak model into a frontier model.

It also does not replace:

- background knowledge
- external evidence
- experiments
- rigorous domain science

What it can do is improve a model's effective intelligence on structured reasoning tasks by improving:

- representation choice
- route selection
- controller detection
- transfer quality
- failure-boundary awareness
- explanation legality

So the honest claim is not:

- "it makes any model generally genius"

The honest claim is:

- "it can materially improve a model's effective reasoning on certain classes of problems by giving it a better cognitive scaffold"

## 7. Difference From Generic Metacognition

Many meta-reasoning frameworks teach:

- reflect
- challenge assumptions
- verify
- branch
- compare options

Those are useful.

But this project is not only about metacognition.

Its special emphasis is on mechanism-grounded operations such as:

- push to boundary
- collapse by symmetry
- search for a conserved or aggregate quantity
- replace the full object with the decisive relation
- read the target from a lower-dimensional slice

So the difference is:

- generic metacognition asks whether the model is thinking carefully
- this system asks what deep operation would actually shrink the problem

## 8. Difference From Existing Auto-Research Systems

Auto-research systems usually focus on:

- retrieval
- paper reading
- hypothesis drafting
- experiment or simulation loops
- report generation

This project is different.

Its intended role is earlier and deeper in the pipeline:

- compress the question
- identify the controlling mechanism
- generate multiple smaller candidate models
- expose likely decisive readouts
- design cheaper falsification paths

So this library should eventually integrate with auto-research systems, but should not be confused with them.

Clean role split:

- this project: compression and mechanism primitives
- literature systems: evidence retrieval
- modeling systems: formalization and simulation
- falsification systems: attack the weakest seam
- memory systems: promote surviving mechanisms

## 9. Honest Research Ambition

The long-range ambition of the project is serious:

- subject-level dimensional reduction
- cross-domain dimensional reduction
- research-transfer dimensional reduction
- multi-route hypothesis generation and elimination

But the project must not claim:

- dimensional reduction alone solves cancer
- a skill library replaces scientific institutions
- reasoning skill alone removes the need for evidence

The honest research value is:

- shrinking false search space
- improving mechanism selection
- surfacing decisive readouts
- helping design low-cost falsification
- making explanation and hypothesis quality more transferable

That is already a high bar.

## 10. The Three-Part Defensive Moat

One possible long-term moat of the system is the coexistence of three layers:

1. `Skill layer`
   - reusable compression primitives
2. `Self-verification layer`
   - every fast route must answer `why is this legal?`
3. `Socratic layer`
   - learners and models can interrogate the hinge until the mechanism becomes explicit

If later connected to autonomous research loops, a fourth layer becomes possible:

4. `Auto-research layer`
   - multi-route evidence gathering, elimination, and mechanism promotion

The moat is not any one layer alone.
It is the disciplined coexistence of all of them.

Underneath them sits an even deeper layer:

- an exploration substrate
- a mechanism-sensitive cognitive layer
- a reusable set of operators for seeing controllers, boundaries, invariants, and compression opportunities

And the project should increasingly move toward:

- autonomous mechanism routing

meaning:

- the model detects, matches, compares, and audits candidate reductions with less human move assignment

## 11. Writing Discipline

When describing the project publicly or internally, prefer language like:

- improves effective reasoning on some classes of problems
- teaches mechanism-sensitive compression
- improves representation choice and route selection
- promotes transferable primitives rather than local tricks
- supports research by reducing false search space and clarifying decisive mechanisms

Avoid or heavily qualify language like:

- solves any hard problem quickly
- makes weak models fully smart
- directly cures cancer
- replaces scientific validation

The stronger the ambition, the more precise the wording must become.

## 12. Final Identity Test

If future work makes the system better at:

- finding controllers
- compressing lawfully
- naming legality
- exposing failure boundaries
- transferring mechanisms across surfaces

then the system is becoming more itself.

If future work only makes it better at:

- sounding clever
- giving shorter answers
- collecting tricks
- mimicking insightful tutoring

then the system is drifting away from itself.
