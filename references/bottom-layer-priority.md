# Bottom-Layer Priority

This file defines a package-level preference:

- if a route has not yet plausibly reached the deepest useful controller, the package should not let middle-layer success feel fully settled

Its purpose is:

- keep the package oriented toward the most fundamental useful carrier
- without prescribing how the model must descend

## Core Claim

The package should distinguish between:

- locally good structure
- deepest useful structure

These are not always the same.

The package must not reward them equally.

## What This Means

If a route reaches:

- one elegant middle layer
- one helpful symmetry picture
- one useful rewrite

that may still be insufficient if:

- a more fundamental shared controller still plausibly exists

The package should therefore preserve pressure against:

- treating middle-layer success as final too early

## What The Package May Do

The package may preserve:

- bottom-layer priority pressure
- remaining fake burden pressure
- carrier-integrity pressure
- weak coexistence of deeper candidates

The package may not:

- prescribe the exact next deeper move
- prescribe a fixed shell taxonomy
- prescribe a recursive descent program

## Relation To Skill Combination

Bottom-layer priority matters because:

- if the route is not yet deep enough, one skill may not be enough
- nearby candidates should not die too early

But the package must still not decide in advance:

- whether the final route is one skill or several

That remains model-side.

It must not convert that preference into a descent script.
