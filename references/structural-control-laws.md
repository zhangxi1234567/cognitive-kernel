# Wu Boshi Structural Control Laws

This file defines the structural laws that explain **why** many visible reduction methods work.

These laws sit between:

- abstract logic laws
- concrete cognitive moves
- named method families

They answer questions like:

- why does symmetry often give extremum?
- why does completion help?
- why does a limit expose the skeleton?
- why can a picture beat a formula?

## 1. Balance Law

When roles are equal and the target reacts to imbalance, the balanced configuration is often the decisive one.

This explains:

- symmetry gives extremum
- equal-case first
- equilateral / isosceles / centered placement guesses

## 2. Straightening Law

A complicated route often becomes solvable after it is straightened into:

- a line
- a shortest path
- one direct displacement
- one monotone axis

This explains:

- process -> static path
- broken line -> straight line
- many local moves -> one direct vector or one support line

## 3. Completion Law

An incomplete object is often harder than the whole simple object it came from.

So the right move is:

- complete the symmetry
- complete the missing focus
- complete the box / rectangle / whole figure

This explains:

- completion methods
- hidden rectangle / midpoint / parallelogram recognition
- full-object restoration before readout

## 4. Degeneration Law

The edge case is often structurally simpler than the interior.

This explains:

- tangency
- coincidence
- endpoint / vertex
- infinite direction
- asymptotic picture

Edge cases work because they remove degrees of freedom and expose the hinge.

## 5. Projection Law

A high-dimensional object is often controlled by one lower-dimensional shadow.

This explains:

- surface -> line
- line -> point
- solid -> slice
- object -> relation
- vector/dot product -> one projection

## 6. Conserved-Burden Law

What matters is not the original object but the burden that must be carried.

If another object carries the same burden more cleanly, switch to it.

This explains:

- constructive surrogate
- mean-value model
- auxiliary function
- same-function comparison

## 7. Machinery-Fit Law

Every target has a shape.
Every tool also has a shape.

If the tool produces the wrong kind of information, the route must change.

This explains:

- symmetrize the target
- set but do not solve
- when the standard route breaks
- switch from Vieta / coordinates / trees to a better-fitting machine

## 8. Anchoring Law

A moving family becomes manageable once one fixed anchor is found:

- fixed point
- fixed line
- center
- midpoint
- baseline
- bottleneck metric

This explains:

- fixed-object subsystem
- landmark geometry
- starting-line patterns

The revealing-case layer explains why:

- coincidence
- tangency
- zero / infinite slope
- symmetric placements

can expose the anchor faster than full-family tracking.

## 9. Compression-Readout Law

Once the object is compressed enough, the final step is often not derivation but readout.

This explains:

- direct count
- read-not-solve
- special ratio direct readout
- eccentricity direct readout
- space diagonal readout

## 10. Scale-Reasonableness Law

Bad routes often expose themselves before proof through:

- wrong unit type
- wrong magnitude
- impossible geometry
- impossible probability

This explains:

- dimension and scale sanity
- option elimination by plausibility
- early route rejection

The deeper logic is:

- every valid route must stay inside the object's compatibility envelope
- type, scale, and boundary mismatches are often enough to kill a route before detailed proof

## 11. Layer-Ownership Law

A route only counts as reduced if the lower layer actually owns the decisive hinge.

This explains:

- low-floor gate
- no hidden formal imports
- lower-level executability

## 12. Play Law

Solving feels like play when:

- the burden has been localized
- the next move is visible
- the control object is small
- the seal is local

This explains why reduction feels empowering instead of exhausting.

## 13. Shell-Translation Law

A new shell should be translated back into an older structural object before heavy solving begins.

This explains:

- new-definition logic
- definition reading
- tiny examples as shell checkers
- translation back to old mathematics

## Hard Rule

Future additions should not only ask:

- which method is this?

They should also ask:

- which structural control law is this surfacing?
