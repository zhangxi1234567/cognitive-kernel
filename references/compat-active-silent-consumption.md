# Compat-Active Silent Consumption

This file defines the compatibility-active form of silent runtime consumption.

Its purpose is:

- make compat-only execution rely more on live field consumption than on visible reasoning surface
- move the package closer to host-native feel without adding workflow control

## Core Claim

Compat-only execution should already feel more alive because fields are:

- consumed
- not merely present

That means compat-only should increasingly rely on silent effects such as:

- warming
- cooling
- settlement restraint
- inward continuation
- readout readiness
- anti-regrowth restraint

Not on:

- visible process narration
- explicit internal step display

## What This Means In Compatibility Form

At the compatibility-active level, silent consumption means:

- runtime state changes route pressure while staying non-binding
- route memory changes later competition texture while staying non-replay
- active fields shape the route without becoming commands

This is not yet:

- full host-native execution

But it is already more than:

- a descriptive compatibility shell

## Allowed Compatibility Pressure

The compatibility layer may already preserve and consume:

- `deeper_carrier_coherence`
- `inward_continuation_pressure`
- `target_complete_readiness`
- `direct_readout_readiness`
- `anti_regrowth_pressure`

These are enough to support:

- weak silent consumption

without:

- visible workflow

## What Must Stay Forbidden

Do not turn silent consumption into:

- “apply this field now”
- a visible runtime checklist
- a state-machine solve order
