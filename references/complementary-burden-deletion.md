# Complementary Burden Deletion

This file defines the project's positive account of complementary burden deletion.

Its purpose is:

- explain why several local mechanisms may matter together without becoming a playlist
- distinguish complementary deletion from duplicated description
- preserve the reason composition can become genuinely sharper than any single move

## Core Claim

Complementary burden deletion happens when:

- one mechanism removes one real obstruction
- another mechanism removes a different remaining obstruction
- the route becomes materially thinner only because both burdens are reduced

It is not:

- several mechanisms restating the same local insight
- rhetorical layering around one already sufficient move

## Complementary vs Duplicative

A local mechanism pair is complementary when:

- each changes what remains alive after the other acts
- each improves carrier quality in a non-identical way
- together they reduce burden more than either could alone at that stage

A local mechanism pair is duplicative when:

- one adds explanation but not deletion
- one confirms what the other already settled
- the pair does not materially change route size or stability

## Host Constraint

The host may preserve:

- whether nearby mechanisms seem burden-complementary
- whether one mechanism has already made the other redundant
- whether the local shell still contains distinct unresolved burden

The host must not preserve:

- a reusable pairing catalogue
- a preferred mechanism duet by topic
- a scripted expansion from one mechanism into several

## Final Rule

The project should allow several local mechanisms to matter together only when their burden deletion is truly complementary.

It must not convert complementarity into a reusable skill-combination recipe.
