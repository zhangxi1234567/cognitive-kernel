# Resonance Cooling

This file defines the project's positive account of cooling.

Its purpose is:

- describe when local plurality has stopped being informative
- preserve natural collapse toward one thinner carrier
- prevent resonance from turning into decorative depth performance

## Core Claim

Cooling is not forced simplification.

Cooling is:

- local plurality losing informational yield
- nearby mechanisms no longer deleting distinct surviving burden
- one carrier becoming sufficient without external lock-in

It is not:

- “always reduce to one move quickly”
- “always keep several moves alive for depth”

## Informative Cooling

Cooling is informative when:

- one carrier has absorbed the useful burden deletion of the local field
- nearby mechanisms have become redundant rather than suppressed
- keeping plurality alive would now add description more than information

Short version:

- informative cooling makes the route thinner

## Decorative Cooling Failure

Cooling fails when:

- plurality is killed early only for neatness
- plurality is kept alive late only for richness
- the field is managed performatively rather than allowed to thin naturally

## Host Constraint

The host may preserve:

- whether plurality still changes the carrier choice
- whether nearby mechanisms still delete distinct burden
- whether one carrier now seems sufficient

The host must not preserve:

- a required cooling threshold
- a fixed “keep N candidates” rule
- a reusable cooling script

## Final Rule

The project should allow local plurality to cool naturally when it stops earning its cost.

It must not convert cooling into a route-management workflow.
