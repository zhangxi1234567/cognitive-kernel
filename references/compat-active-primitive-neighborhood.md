# Compat-Active Primitive Neighborhood

This file defines the compatibility-active form of primitive-neighborhood activation.

Its purpose is:

- make compat-only execution more likely to light up several nearby lower-level primitives from one live structure
- preserve primitive plurality without turning it into a primitive checklist

## Core Claim

Compat-only execution should already allow one structure to weakly illuminate a neighborhood such as:

- boundary
- limit
- special value
- symmetry
- direct comparison
- projection

depending on what local burden remains.

This is not:

- “use all of them”
- “try them in order”

It is:

- neighborhood availability

Interpretation rule:

- a primitive neighborhood is a local affordance field
- not a recommendation to expand into a primitive checklist

## What This Means In Compatibility Form

At the compatibility-active level, primitive-neighborhood activation means:

- one live carrier can make several nearby primitives visible at once
- those primitives can remain weakly available without all becoming full routes
- the eventual winner may absorb useful pressure from that neighborhood

## Allowed Compatibility Pressure

The compatibility layer may already preserve:

- `primitive_neighborhood_breadth`
- `primitive_field_visibility`
- `primitive_reachability_pressure`

These are enough to support:

- weak primitive-neighborhood life

without:

- primitive choreography

## Final Rule

Compat-only execution should already allow one structure to light up a small nearby primitive neighborhood.

It must not expose that as a primitive checklist.
