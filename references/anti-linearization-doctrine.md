# Wu Boshi Anti-Linearization Doctrine

This file protects the project from one subtle failure mode:

- turning a deep cognitive system back into a visible linear workflow

As the library grows richer, it becomes tempting to rewrite it into:

1. do controller-world detection
2. do operator selection
3. do primitive selection
4. do verification

That may be useful as a map.
It is dangerous as a mandate.

If the project becomes a forced visible sequence, it will lose the very freedom it is trying to create.

## 1. Core Rule

The project's layered structures are:

- cognitive topology
- routing space
- internal organization

They are not:

- a rigid linear execution script
- a mandatory visible checklist
- a fixed user-facing step order

Short version:

- map, not script

## 2. Why This Matters

The point of the library is to help the model:

- see controllers
- sense possible operators
- compare multiple routes
- jump toward the most revealing structure

Strong reasoning often does not feel linear from the inside.

It may:

- leap to a boundary
- jump straight to symmetry
- notice an aggregate first
- move from a primitive directly to a world-level reinterpretation
- switch worlds midway

That flexibility must be preserved.

## 3. Layers As Implicit Space

The layers:

- controller worlds
- operators
- primitives
- laws
- mechanisms

should be treated as an implicit cognitive space.

Their purpose is to:

- stabilize meaning
- enable free matching
- allow internal organization
- improve memory and transfer

Their purpose is not to force every solve into explicit stage-by-stage narration.

## 4. Acceptable Uses Of Layering

Layering is good when it is used to:

- organize the ontology
- train internal recognition
- store memory cleanly
- compare candidate routes
- explain after the fact why a route worked

Layering becomes harmful when it is used to:

- force one visible order on all problems
- make the model narrate artificial intermediate steps
- replace intuition with compliance
- convert free operator composition into scripted behavior

## 5. The Right Kind Of Freedom

The project does not want chaos.
It also does not want rigid sequencing.

It wants:

- freedom with structure

That means:

- the system has a deep map
- but movement inside the map is flexible
- the model may enter from world, operator, primitive, or even a direct readout intuition
- after entry, it may move nonlinearly until a truthful compression stabilizes

## 6. Internalization Rule

The more mature the system becomes, the less these layers should need to appear explicitly in user-facing behavior.

The desired end state is:

- the model has internalized the topology
- the model moves fluidly within it
- the model does not need a human to announce the lane
- the model does not need to visibly recite the map before solving

So success is not:

- better visible compliance with the ontology

Success is:

- deeper invisible use of the ontology

## 7. Writing Rule

When documents use sequences such as:

- world -> operator -> primitive

they should be read as:

- structural decomposition
- memory organization
- routing abstraction

not as:

- mandatory solve order

If a reader can mistake the sequence for a rigid solve script, the wording should be tightened.

## 8. Final Rule

This project should never force the model to think in a straight line merely because the documentation is organized in layers.

The layers exist so thought can become freer without becoming shapeless.
