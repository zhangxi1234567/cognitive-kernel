# Wu Boshi Derivative Route Superfamily

This file is the consolidated superfamily surface for the derivative-heavy route line.

Its purpose is:

- unify derivative route taxonomies, surrogate construction, mean-value models, graph behavior, and motion bridges
- separate stable route logic from still-unresolved public five/six-type member lists

## Core Principle

Do not treat derivative problems as one default symbolic procedure.

Ask first:

- what route family owns this problem?
- what control object makes the derivative burden smaller?
- what lower-floor bridge makes the route executable?

## The Main Internal Layers

### 1. Route-Taxonomy Layer

Use:

- `Derivative-Series Route Taxonomy`
- `Derivative-Type Router`

Purpose:

- classify the task before doing derivative work

### 2. Surrogate / Bridge Layer

Use:

- `Constructive Surrogates`
- `Mean-Value Models`
- `Math-Physics Bridges`

Purpose:

- replace the raw derivative flow with a better control object

### 3. Local-Behavior Layer

Use:

- `Starting-Line Patterns`
- `Shape-Behavior Microclusters`
- `Same-Function Comparison`
- `Isomorphic Shape Families`

Purpose:

- read the function by local behavior, graph shape, and inherited mother behavior

### 4. Public-Taxonomy Registry Layer

Use:

- `public-derivative-taxonomy-registry.md`

Purpose:

- store unresolved but confirmed public five/six-type route banks

## What This Superfamily Already Knows

- how to classify derivative routes
- how to replace brute derivative flow by smaller bridge objects
- how to use motion, start conditions, and graph behavior as lower-floor controllers

## What Still Depends On More Source Access

- exact public five/six-type enumerations
- transcript-level definitions for each public type
- stronger member-by-member mapping from titles to stable route cards

## Hard Rule

Do not confuse:

- “we do not yet know the exact public list”

with:

- “the derivative route logic is not integrated”

The route logic is already part of the library.
