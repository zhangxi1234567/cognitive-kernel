# Wu Boshi Style Expansion

This file defines how the skill should grow new methods from style, not just store old tricks.

## Core Rule

The skill must not treat the current method list as closed.

A Wu Boshi-style skill should repeatedly ask:

- what is the deeper move behind this trick?
- what nearby problems share that move?
- what new variant appears if I change the object, target, representation, or scale?

That is how one trick becomes a family.

## Style To Method Expansion

Start from the style, not the chapter.

Wu Boshi style repeatedly prefers:

- strip the shell
- lower the grade
- find the smallest controlling structure
- replace full solving with comparison / counting / balancing / anchoring / compression
- prefer one decisive observation over complete procedural grind

From this style, new methods should be generated by asking:

### 1. Change the object

If the current trick works on:

- numbers

ask whether it also works on:

- vectors
- functions
- sets
- geometric objects
- states
- particles

### 2. Change the target

If the current trick solves:

- exact value

ask whether the same trick also solves:

- comparison
- sign
- existence
- range
- fixed point / fixed line
- count
- best / worst case

### 3. Change the representation

If the current trick was found in:

- algebra

ask whether it becomes cheaper in:

- picture
- table
- graph
- flow
- balance sheet
- permutation
- area
- projection

### 4. Change the scale

If the current trick works for:

- a small toy case

ask:

- what stays true at larger scale?
- what aggregate replaces local detail?
- what quantity becomes the new bottleneck?

### 5. Change the subject

If the current trick is from:

- school math

ask whether the same skeleton appears in:

- olympiad math
- university math
- physics
- chemistry
- debugging
- planning

## The Expansion Question

For every method or primitive, ask:

1. What structure did this method really exploit?
2. What other objects share that structure?
3. What other targets can be solved if I keep the same structure?
4. What representation makes this structure even cheaper?
5. Where does the method break?

If you can answer 1-4 clearly and 5 honestly, you have a new method family, not just one trick.

## Allowed Expansion Directions

Good expansions:

- `compare without calculating` -> compare size, sign, threshold, monotonicity, best option
- `matching instead of probability` -> assignment, derangement, schedule, seating, tournament pairing
- `container to cross-section` -> packing, tangency, shortest distance, visibility slice, flow bottleneck slice
- `common-value compression` -> shared output, hidden parameter, one-control-variable reduction
- `permutation recognition` -> row-column choice, nonattacking placement, bijection, one-to-one encoding

Bad expansions:

- promoting a lucky numerical coincidence
- copying phrase style without preserved mechanism
- generalizing from one diagram that only works because the picture is clean
- forgetting the failure mode

## Expansion Must Preserve Truth

Every new method generated from style must still pass:

- legality
- transferability
- failure-mode clarity
- minimal verification

If it cannot, it stays a local trick.
