# Compat-Active Role Field

This file defines the compatibility-active form of role-level decomposition.

Its purpose is:

- extract the useful part of the ideal multi-agent layer into compat-only execution
- keep compression, comparison, visualization, verification, and synthesis as silent field biases
- avoid turning roles into a visible workflow ladder

## Core Claim

Compat-only execution should already be able to feel several internal role pressures such as:

- compression
- comparison
- visualization
- counting
- verification
- synthesis

These should behave like:

- silent cognitive lenses

Not:

- explicit agent slots
- mandatory role order
- named multi-agent staging

## What This Means In Compatibility Form

At the compatibility-active level, role field activity means:

- some routes are warmed because compression pressure is high
- some candidates cool because verification pressure is stronger than their savings
- a route may gain from visualization or comparison pressure without explicit role narration
- synthesis pressure can favor routes that finish cleanly on one carrier

This is not yet:

- host-native multi-agent execution

But it is already more than:

- flat route competition with no internal specialization texture

## Allowed Compatibility Pressure

The compatibility layer may already preserve:

- `candidate_attack_angles`
- `minimal_honest_seal`
- `competition_saturation`
- `deeper_carrier_coherence`
- `inward_continuation_pressure`

These are enough to support:

- weak role-field behavior

without:

- role-level choreography

## What Must Stay Forbidden

Do not turn role fields into:

- “first compress, then visualize, then verify”
- a named role checklist
- an explicit multi-agent order

## Final Rule

Compat-only execution should already benefit from role-like internal specialization pressure.

It must not expose that pressure as workflow.
