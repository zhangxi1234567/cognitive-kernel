# Compat-Active Resonance Cluster

This file defines the compatibility-active form of resonance clusters.

Its purpose is:

- make compat-only execution more likely to keep several nearby angles alive as one local field
- preserve natural clustering without forcing explicit multi-skill choreography

## Core Claim

Compat-only execution should already allow:

- several nearby candidates to feel locally related
- those candidates to strengthen each other's visibility
- the cluster to persist briefly before natural consolidation

This is stronger than:

- isolated weak alternatives

But weaker than:

- mandatory multi-route display

## What This Means In Compatibility Form

At the compatibility-active level, a resonance cluster means:

- nearby candidates are not merely coexisting by accident
- they are alive together because they illuminate overlapping surviving burden
- one later route may absorb the useful burden deletion of the cluster

This is not:

- “use these together”
- “compare them in order”

Interpretation rule:

- resonance is a local visibility effect, not a managed combination plan
- the cluster may cool before becoming any explicit multi-primitive route

## Allowed Compatibility Pressure

The compatibility layer may already preserve:

- `resonant_angle_pressure`
- `coactivation_density`
- `consolidation_gravity`

These are enough to support:

- weak resonance-cluster life

without:

- visible orchestration

## Final Rule

Compat-only execution should already allow local resonance clusters to form and cool naturally.

It must not expose them as a scripted combination plan.
