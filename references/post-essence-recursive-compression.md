# Wu Boshi Post-Essence Recursive Compression

This file defines a compatibility-facing rule:

- catching the essence is only the first honest compression

Its purpose is:

- keep the package from rewarding a correct mid-route insight as if it were full reduction
- preserve pressure toward continued burden deletion
- do so without turning continuation into a visible recursive script

## Core Claim

The project should reject this failure mode:

- the model catches the essence
- the controller is named correctly
- the route then settles into ordinary completion
- avoidable burden survives because the first insight felt final

That is not the target.

The target is:

- essence changes the route
- the new carrier stays inspectable
- nearby lawful reductions remain able to surface
- sealing happens only when important fake burden is gone

Short version:

- essence is the beginning of deeper compression pressure, not permission to stop

## What This File Does And Does Not Mean

This file does mean:

- the route should remain open to further honest shrinking after the first real controller is found

This file does not mean:

- externally tell the model to recurse
- externally tell the model what question to ask next
- externally require named skill chaining
- externally require a second or third compression step

The project wants:

- internally continued compression

not:

- externally staged deepening

## What Post-Essence Compression Really Means

Post-essence compression means:

- the first reduction changes the carrier
- that new carrier may still be too thick
- if a smaller truthful carrier is now visible, the route should remain able to move again
- if no important fake burden remains, the route may honestly seal

This is not mechanical repetition.

It is:

- renewed openness to smaller truthful structure

## Canonical Failure Modes

### Failure A: Essence Then Ordinary Solve

Shape:

- the controller is named correctly
- the remaining burden is paid by standard flow

Problem:

- the first compression happened
- later compression never became live

### Failure B: Essence Then Decorative Skill Mention

Shape:

- one good skill is named
- but no deeper carrier shift follows

Problem:

- the route sounds aligned
- the route does not become more ruthless

### Failure C: Essence Then Premature Seal

Shape:

- the route finds one plausible reduced picture
- then settles before the burden is truly small

Problem:

- middle-layer success is mistaken for bottom-layer hit

## What The Compatibility Layer May Preserve

The compatibility layer may preserve weak signals such as:

- `remaining_fake_burden`
- `bottom_layer_priority`
- `composite_shell_risk`
- `question_emergence_pressure`
- `primitive_reachability_pressure`
- `resonant_angle_pressure`

These may jointly support:

- natural renewed scrutiny
- natural multi-skill composition
- natural collapse to one route when deeper competition is no longer useful

They must not become:

- a fixed continuation ritual
- a fixed "go deeper" command
- a forced post-essence checklist

## Evaluation Rule

After the solver catches the essence, ask only:

- did the route actually become smaller afterwards?

Useful signs include:

- a lower carrier took over
- nearby candidates briefly became live
- one candidate or composition deleted additional burden
- the final seal became cheaper because of later compression

If none of that happened,
the project should suspect under-use of the skill system even if the wording sounds aligned.
