# Anti-Proceduralization

This file defines a hard rule:

- the compatibility layer must not proceduralize the model's thought

Its purpose is:

- stop the package from degrading into a thought program
- preserve self-driven exploration

## Core Claim

The package may preserve:

- weak pressure
- weak competition
- weak memory
- weak caution

The package may not preserve:

- a thought program

## What Counts As Proceduralization

Proceduralization includes:

- fixed solve ladders
- fixed recursion ladders
- fixed comparison order
- fixed role order
- fixed “ask this question next” prompts
- fixed “keep these many candidates alive” rules
- fixed “now narrow to one route” rules

Even if these are meant to help, they still replace:

- model-side route formation

with:

- package-side thought control

## What Is Still Allowed

Allowed:

- signals that more than one route may matter
- signals that the current carrier may be premature
- signals that a route may still be too thick
- signals that nearby weak candidates should not die too early

These are:

- pressures

not:

- procedures

Deproceduralization is not shorter choreography.

It is removal of host-owned step authority.

## Why This Matters

The project does not want to teach:

- how to obey a deeper-looking checklist

It wants to improve:

- how the model naturally forms, keeps, and settles routes

If the package begins telling the model exactly how to think, it has failed,
even if the resulting behavior looks more structured.

The package may shape the field.

It must not become the driver.
