# Composition Necessity

This file defines the project's positive account of when composition is necessary.

Its purpose is:

- distinguish genuine multi-mechanism need from decorative plurality
- preserve composition only when one mechanism alone does not delete the live burden honestly
- prevent composition from becoming a display of richness

## Core Claim

Composition is not necessary because several mechanisms are available.

Composition is necessary only when:

- the live shell carries more than one distinct kind of surviving burden
- no single mechanism currently deletes those burdens honestly by itself
- multiple local mechanisms are each removing different real obstructions

It is not:

- a proof of sophistication
- a reason to keep several mechanisms alive by default

## Necessary Composition

Composition is genuinely necessary when:

- one mechanism exposes a thinner carrier
- another mechanism removes a different remaining obstruction inside that carrier
- a further mechanism provides the cheapest honest seal

Short version:

- composition is necessary when burdens are complementary, not duplicated

## Decorative Composition

Composition is decorative when:

- the same burden is being described repeatedly by nearby mechanisms
- one mechanism is already sufficient but others are retained for richness
- plurality no longer changes carrier choice, route stability, or seal economy

## Host Constraint

The host may preserve:

- whether one mechanism alone still appears insufficient
- whether different local mechanisms are deleting distinct burden
- whether composition has become redundant after one carrier sharpens

The host must not preserve:

- a required bundle size
- a reusable composition recipe
- a fixed rule for when composition must expand

## Final Rule

The project should allow composition only when it is burden-necessary.

It must not convert necessity into a reusable combination workflow.
