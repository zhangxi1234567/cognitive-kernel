# Wu Boshi Function As Relation Carrier

This file defines the bottom-layer view:

- a function is not first a formula
- it is a rule that assigns outputs to inputs
- the true low-floor carrier is the set of input-output relations

## Core Rule

When a function problem looks abstract, ask:

- what points or input-output pairs must satisfy the rule?
- what relation between outputs survives all allowed inputs?
- can a few strategically chosen inputs expose the mechanism?

Then compress:

- function shell -> relation carrier on points

## Bottom-Layer Logic

- a function is controlled by what it does to inputs
- formulas are one shell for that rule
- values, symmetry relations, shifts, and recurrences are more primitive than symbolic appearance

This is why:

- assignment works
- special values work
- symmetry in `f(x), f(-x), f(x+a)` works
- periodicity and parity can be read from input-output relations

## What This Explains

- abstract function -> pick probe inputs
- new-definition function -> read the rule first
- graph-like behavior -> read values and transformation law
- function equations -> treat them as relational constraints, not only algebraic expressions

## Good Uses

- function equations
- abstract function multiple choice
- parity and periodicity
- range / monotonicity by probe points

## Failure Boundary

Do not overfire assignment when:

- the chosen points do not stress the real mechanism
- local sample values cannot certify the global claim
- the function class is too broad for sparse probing alone

## Transfer

- school math: assign special `x` values
- university math: probe test points, basis inputs, canonical elements
- research-style transfer: treat a model as an input-response mechanism before worshipping the formula
