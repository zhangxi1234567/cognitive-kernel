# Nativeization Without Workflow

This file defines the hard rule for making the package feel more native

without:

- turning it into workflow
- turning it into visible guidance
- turning it into hidden prompt choreography

## Core Claim

The project should become more native through:

- silent field consumption
- stronger host-side modulation
- richer behavior-level convergence

Not through:

- more explicit solve order
- more internal monologue templates
- more visible decomposition steps

## What Nativeization Should Mean

Nativeization should increasingly mean:

- the host consumes the field
- the host warms and cools candidates silently
- the host restrains premature settlement silently
- the host supports inward continuation and direct readout silently
- the host resists regrowth silently

This is:

- deeper internalization

Not:

- more instructions

## What Must Stay Forbidden

Do not approximate nativeization by adding:

- forced question prompts
- visible “go deeper” reminders
- forced role ladders
- forced operator ladders
- mandatory multi-route narration

Those are not nativeization.

They are:

- workflow substitution

## Final Rule

If nativeization makes the package more alive and less visible, it is progress.

If nativeization makes the package easier to narrate and easier to command, it is regression.
