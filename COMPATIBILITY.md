# Wu Boshi Compatibility Boundary

This file is intentionally small.

Its purpose is not to teach the host how to solve.

Its purpose is only to keep the host from distorting the package into visible control.

The package should be read as a cross-domain cognition layer, not as a math curriculum.

## What This Layer Is

This layer is:

- a thin compatibility boundary
- a non-workflow distortion guard
- a small mechanism-first bias surface

This layer is not:

- the ideal layer itself
- a solve script
- a routing table
- a skill menu
- a decomposition ladder
- an answer template

## Hard Negative Constraints

The host must not turn the package into:

- explicit step order
- explicit route order
- explicit skill order
- explicit composition recipe
- explicit role choreography
- explicit checklist execution
- explicit output skeleton

- explicit reference-driven solve lookup

The host must not enlarge the compatibility layer in order to imitate native behavior.

If a stronger native feel requires more visible packaging, that is regression.

## Allowed Effect

The compatibility layer may only do three things:

- reduce ordinary-regrowth bias
- make smaller truthful structure easier to notice
- keep visible external control from hardening around the run

In practice this may appear as:

- fake shell pressure being easier to feel
- a smaller live carrier becoming easier to trust when it really carries the task
- brief local coexistence of several lines while one line is not yet enough
- nearby live lines tightening toward one unresolved burden without explicit choreography
- sharper local attack appearing when the remaining burden has become sufficiently cornered
- natural cooling once a line stops carrying real burden
- honest stopping once no live burden remains

Any stronger claim belongs to the ideal native layer,
not to the current compatibility layer.

## Accurate Present-Tense Claim

The project already has:

- a real compatibility boundary

The project does not yet have:

- host-native silent runtime modulation
- perfect host parity across every Codex / Claude Code entry surface

## Current Compatibility Status

Current status is:

- compatible with Codex entrypoints
- compatible with Claude Code entrypoints
- not yet perfect in thinness or host-native absorption

What already works:

- root entrypoints exist for both hosts
- the manifest identity is cross-domain and non-workflow
- runtime, compat, benchmark, and archive layers are now explicitly separated

What still prevents calling it perfect:

- `agents/openai.yaml` is still thicker than the ideal thin runtime surface
- legacy archive gravity still exists under `references/`
- some compatibility doctrine still survives as copied bridge material rather than fully native host behavior

So the honest claim is:

- structurally compatible now
- directionally aligned now
- not yet perfectly nativeized

## Final Rule

This layer is a thin bridge.

It should become smaller as hosts become stronger.

Its reference archive should become less live, not more operational.

See `PACKAGE_MAP.md` for the active split between runtime, benchmarks, and archive.
