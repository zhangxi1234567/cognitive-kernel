# Benchmarks Layer

This directory exists to answer one question:

Does the layer survive contact with real work outside one school-math shell?

## What Belongs Here

- minimal verification doctrine
- cross-domain anchors
- transfer tests
- benchmark notes that measure whether the package remains metacognitive rather than topic-bound
- adversarial false-essence drift tests
- win notes that preserve benchmarked anti-false-essence breakthroughs
- blind elimination tournament protocols for comparing candidate solve shapes under hard pressure

## What Does Not Belong Here

- solve scripts
- topic routers
- method menus
- archive-heavy tactic catalogs

Benchmarks should test whether the laws still hold in math, coding/debugging, and research.

They should not reintroduce the very workflow shell the runtime is trying to thin.

They should also test whether the runtime can reject:

- elegant but local impostors
- deeper-but-irrelevant stories
- false symmetry
- retention layers mistaken for validity owners
- one-site fixes mistaken for complete closure

Relevant files:

- `MINIMAL_VERIFICATION_PROTOCOL.md`
- `CROSS_DOMAIN_ANCHORS.md`
- `MULTI_AGENT_BLIND_ELIMINATION_PROTOCOL.md`
