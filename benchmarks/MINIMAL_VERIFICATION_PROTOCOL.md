# Minimal Verification Protocol

This is not a workflow.

It is a thin refusal to let a beautiful claim pass without the cheapest honest pressure.

Use it when a mechanism feels clean, explanatory, or "obviously right" and you want to know whether it survives first contact with reality.

## Core Move

Do not ask first whether the story is elegant.

Ask what would fail first if the story were wrong.

Prefer checks that are cheaper than continued explanation:

- boundary behavior
- edge or degenerate cases
- invariant preservation
- dimension or unit sanity
- one concrete counterexample attempt
- local consistency with nearby claims, code paths, or observed facts
- one direct witness for each proposed extra repair surface
- one check that the proposed essence is not merely a local family pattern

If one of these breaks the claim, the beauty was carrying fake certainty.

If they all survive, confidence may rise, but only in proportion to what was actually stressed.

## Pressure, Not Procedure

Usually very few checks are enough if they actually touch the live seam.

Pick the seam most likely to crack:

- At a boundary, does the rule still say something coherent?
- In an extreme or degenerate case, does the controller remain the same or reveal the lie?
- If a quantity changes name but not unit, is the mechanism still dimensionally lawful?
- If the claim predicts behavior, can one hostile example be constructed cheaply?
- If the explanation is global, does a local slice still agree with it?
- If a second fix location is being proposed, what observation forces that second location instead of merely permitting it?
- If the explanation is built on one named family or one residue class, what deeper validation law would still matter if that naming shell were removed?
- If the controller is claimed to live at one abstract boundary, which sibling entry points cross that same boundary under the same invariant?
- If a higher-level explanation is being proposed, what concrete witness failed to be explained at the lower already-evidenced layer?
- If a lower-level fix is being proposed, does the same proved invariant remain half-open at sibling entry points on that boundary?
- If the first exposed function has been found, what adjacent semantic ring would still remain open: AST, stringify, ID, signature, rendering, or shared lexical law?
- If a sibling path, helper extraction, or neighboring domain is being proposed, what independent witness says the same law actually crosses there too?
- For crash-class failures, which exact interface, storage edge, ownership edge, or mutation path is the witness actually touching?
- If a deeper upstream story is being proposed, what present-task closure does it buy that the nearer boundary did not already provide?
- If the bug is about a carried label, freq, dtype, or attached meaning, which boundary finally authorized that thing to remain on the result?
- If one layer produced or converted the result and another later pinned, preserved, inferred, or validated metadata on it, which layer still had real discretion over the wrong outcome?
- If one layer kept or reattached the metadata, which later rule or function still owned the exact test of whether that metadata was actually valid for this result?
- If one layer makes the visible distortion legible, what later layer still decides whether that distortion actually survives onto the finished result?

The aim is not exhaustive validation.

The aim is to prevent decorative mechanism language from outrunning contact with truth.

## Runtime-Adjacent Use

When this sits near runtime or evaluation, the check should stay close to execution:

- a smallest failing input
- a case at a branch boundary
- a conserved quantity before and after a transformation
- a unit-bearing quantity whose scale should not silently drift
- a neighboring implementation or observation that should agree if the story is real

Do not grow a harness when a direct witness is available.

Do not write ten tests to avoid looking at the one case most likely to falsify the claim.

Do not widen a patch because a nearby branch "might also matter" unless one cheap witness reaches that branch too.

Do not widen from one repaired ring into the next merely because the expansion looks architecturally tidy.

One more sibling, one more helper, or one more neighboring domain still needs one more witness.

One more proposed edit means one more burden of proof.

Do not accept a local alias, family, or same-class explanation as essence until one cheap check asks whether the real controller is a broader compatibility or validation rule.

Do not stop at one entry point when the evidence already says the failure belongs to a shared ownership, validation, or bookkeeping boundary; in that case, closure requires either covering or honestly ruling out sibling entry points that carry the same witnessed law.

Do not climb to a cleaner architectural story when a directly evidenced lower boundary already explains the failure and yields a complete fix.

Do not move upstream just because the deeper layer feels more universal.

Move only if the current boundary still leaves the present witness incomplete, ambiguous, or falsely generalized.

Do not confuse inference with authorization.

When one layer guesses, classifies, or proposes a label and another layer decides whether that label is still valid on the final result, the second layer usually owns the bug.

Do not confuse production with validation either.

When one layer makes the new object and another decides what old claims may survive on that new object, the second layer usually owns the bug unless the first layer mechanically forced the answer.

Do not confuse keeping a label with proving it.

If a later function still decides whether the kept label conforms to the exact result values, spacing, anchor, or invariant, that later function is usually closer to the essence.

Do not confuse "lowest" with "complete", and do not confuse "complete" with "highest".

The target is the first layer where the witnessed law closes fully without speculative lift.

If the same witnessed law is still open on the same boundary, verification is not complete no matter how elegant the local patch looks.

## Failure Signal

The protocol has worked if it produces one of these outcomes quickly:

- the mechanism survives a real seam and becomes less hand-wavy
- the claim collapses early and cheaply
- the original explanation is replaced by a smaller, truer one
- an over-wide fix shrinks back to the minimal surface actually required by evidence
- a pretty local explanation gets replaced by a smaller but more general controller
- a one-site fix grows just enough to cover the same proven invariant across sibling entry points
- a too-high explanation collapses back down to the lowest layer that was actually witnessed
- a too-narrow fix grows just enough to close the same witnessed invariant on its true boundary
- a too-deep explanation collapses back to the first boundary that already closes the present witness specifically
- a label-level drift bug collapses to the boundary that had final authority to keep or reject that label on the result
- a translation-induced metadata bug collapses to the boundary that judged whether the translated result could still carry the old metadata claim
- a retention-induced metadata bug collapses one step further to the boundary that owned the exact legality test for the retained metadata

If nothing falsifiable was ever touched, verification did not happen.
