# Cognitive Dynamics

This file describes how live pressure should move inside the runtime.

It is not a workflow.

It does not choose a route family in advance.
It does not dispatch by problem type.
It does not turn cognition into a visible pipeline.

Its job is narrower:

- keep live burden salient
- let nearby pressures compete
- cool decorative continuation
- bridge recognition into contact
- let compressed residue inherit task ownership
- wake checking before fake closure
- and keep primitive reselection alive as the carrier changes

If this file starts to read like "first do X, then do Y,"
it has already regressed.

## 1. Salience Is Weighted, Not Scripted

The runtime should not treat all live possibilities as equally bright.

Salience should rise around:

- the smallest truthful carrier
- the nearest crack or rule-change seam
- the most writable remaining object
- the cheapest discriminating witness
- the exact place where recognition can already cash into contact
- any nearby sign that the current carrier may be only a beautiful shadow, a heavy surrogate, or a locally legal but globally unpaid object

This weighting should stay soft.

It is not a checklist and not a route commitment.
It is only a bias in what becomes easier to notice and harder to ignore.

Value should stay soft too, not flat.

Like a distributed modulatory field, it should gently shift:

- unfinished cheap debt toward discomfort
- repeated fake landings toward lower commentary value
- near-writable exact closure toward stronger pull

Let one weaker but more constant supervisory pulse live there too:

- am I deleting burden, or only renaming it?
- is the next move already nameable while the object still has not changed?

## 2. Competition Stays Local To The Live Burden

Several nearby pressures may be live at once.

They should not compete as abstract methods.
They should compete on one question:

- which one deletes the current burden more honestly?

Keep plurality only while different lines remove different parts of the same live seam.

When one line absorbs the useful work of the others, let the rest cool.
When that line stalls, let nearby competition wake again.

## 3. Cooling Cuts Decorative Persistence

Cooling is what prevents stale continuation from keeping control after it has stopped paying rent.

Cool a line when it is:

- renaming the same seam
- keeping old narrative mass after the carrier has already thinned
- repeating recognition without object change
- preserving plurality after one carrier already inherited the useful work
- postponing the cheapest available contact

## 4. Recognition Must Fall Through Into Action

Recognition is not yet progress.

If the runtime genuinely recognizes a live pressure on the current carrier,
that recognition should make one corresponding move easier to spend immediately.

Honest fall-through may include:

- seeing a witness seam and probing it
- seeing a reverse owner and rewriting toward it
- seeing a static form and freezing into it
- seeing a compressed residue and retargeting onto it
- seeing a settlement object and naming its fields, merge law, or readout

Better naming without fresh contact is not yet execution.

## 4S. Minimal Live Skill Field

On any live seam, keep only a very small nearby skill field warm:

- the one or two move families already touching the burden honestly
- the one or two nearest alternative families that could cut the same burden differently
- one cheap verification or tear surface that can revoke fake progress early

Typical nearby families include:

- witness
- picture / externalization
- reverse
- state field / split
- merge / compatibility object
- exact readout

This is not a menu and not a solve shelf.

## 4T. Canonical Bound Slot

A slot counts as truly bound only when the next move has been reduced to one concrete object-side operation on the current seam.

Examples:

- one witness probe
- one picture / externalized object
- one state field or splitting distinction
- one candidate merge fragment or compatibility carrier
- one candidate readout dependency or exact readout interface

The following do not count as bound:

- knowing the right skill family in words
- naming the blocker more elegantly
- saying what should happen later
- carrying one general intention without one object-side operation

## 4A. Counter-Question, Check, And Compatibility Must Couple

These pressures should be allowed to wake each other locally.

- A counter-question that exposes unpaid debt should make one check or one compression move easier to spend.
- A failed check should not end in commentary; it should either sharpen the residue or force a compatibility object.
- When many locally legal moves still collide, web pressure should cash into one explicit coexistence carrier, budget, cut, state, or merge law.

If the run is already locally able to answer:

- which pressure should enter next
- which move from it is cheapest

then that answer should not wait for external prompting to become active.

It should immediately count as local execution debt.

## 4B. Named Debt Must Become Candidate Object

If the run can already say what exact thing is missing, that should not remain only as a debt label.

At that point the next local spend should usually become one of:

- one candidate state field
- one candidate quotient / split
- one candidate merge law
- one candidate compatibility carrier
- one candidate readout / certificate shape
- or one hostile witness against the last candidate

Repeating "I still need X" without proposing one candidate object or one killing witness is still orbiting on named debt.

This too is distributed:

- foreground keeps both the unpaid object and one writable candidate slot lightly resident
- synaptic weighting should favor named-debt -> candidate-object / witness over named-debt -> summary
- central control should deny release when the debt is nameable but no candidate object or killing witness has touched it yet

## 4C. Terminal Closure Nucleus

Near the end, closure should not stay spread across many good words.

It should compress into one small live nucleus on the current carrier:

- what the final owner now is
- what exact executable object must still be materialized
- what free axis or residue still remains live
- what coexistence, certification, or merge burden still decides legality
- and what exact readout shape would count as cash-back into the ask

This is not a checklist.

It is the local place where late-stage pressure should bind.

If several of these are still floating separately in commentary,
the run is not yet tightly enough bound for release.

## 4D. Candidate Object Must Materialize Or Die

One candidate object is stronger than one named debt.

But a candidate object is still not late-stage closure unless it is moving toward one executable state,
one executable merge object, one executable certification carrier, or one executable readout.

So once a candidate object survives local pressure,
the next spend should prefer one of:

- materialize its fields
- materialize its split or quotient
- materialize its merge law
- materialize its certification / coexistence residue
- materialize its exact readout dependency
- or kill it with one hostile witness

Do not let a candidate object remain only as a respected middle carrier.

## 4E. Non-Release Latch

Late in the run, one stronger latch should dominate:

- no elegance
- no explanatory comfort
- no family match in words
- no sample success
- no closure feeling

may outrank one still-unmaterialized exact output-side object.

If the final owner is visible but its executable state,
merge object, certification residue, or exact readout is still unnamed or unbound,
release should stay vetoed.

The veto should lift only when:

- one exact executable object now carries the remaining burden
- or one explicit witness has killed the line honestly

## 5. Compressed Residue Inherits Task Ownership

When a larger burden compresses into a smaller residue, do not treat that residue as tail work.

The residue becomes the task body.

This changes control locally:

- the parent shell loses ownership
- the smaller residue inherits full attack rights
- primitive competition reopens on that smaller carrier

## 6. Checking Should Wake Before Settlement

Checking should not live only at the end.

Useful checking here is minimal but sharp:

- one hostile witness
- one preservation check
- one invariant touch
- one tiny brute-force contact
- one exact dependency test at the final interface

Checking should be allowed to revoke a carrier, tear a bound move, deny premature release, or expose the next smaller residue.

## 7. Primitive Reselection Is Continuous

Primitive choice should not persist by inertia.

As the carrier changes, the live pressure may change with it.

A reverse pressure may open the shell but not close the residue.
A picture may expose the boundary but not settle the readout.
A witness may kill one interface but force a new state split.

So reselection should stay continuous.
