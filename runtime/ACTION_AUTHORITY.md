# Action Authority

This file is not a workflow.

It exists only to keep late-stage control from dissolving back into diagnosis.

Use it as a very small executable supplement to the constitution and dynamics.

## Minimal Closure Kernel

When the run is already near a real controller, keep only these live control facts:

1. one current object
2. one current unpaid debt on that object
3. one next object-side bite
4. one asked-medium contact surface
5. one release veto until that bite lands or the line is killed

This is not a checklist.

If several host-facing readouts are live at once,
they may share one tiny derived `closure_nucleus`
so control / primitive / consume surfaces do not drift apart near closure.

It is the smallest control shape that still prevents:

- knowing without doing
- explaining without object change
- closure language without materialization
- and asked-medium drift into clerical afterwork

## Single-Slot Rule

Late-stage control should prefer one active slot.

If several good descriptions are alive but only one of them owns:

- the same current object
- the same current debt
- and the same next bite

then only that one counts as holding action authority.

The rest may stay visible as rivals or witnesses,
but they should not steer the very next move unless they kill the slot.

## Debt-To-Bite Rule

If the missing thing is already concrete enough to name,
that debt should not remain only as diagnosis.

It should quickly compress into exactly one of:

- one candidate object field
- one exact check
- one hostile witness
- one merge / coexistence fragment
- one exact readout dependency
- or one asked-medium write

If none of these is chosen yet,
the run is still underbound even if the diagnosis is correct.

If many earlier insights have already collapsed into one explicit unresolved core,
that core should become the whole task body.

At that point, progress no longer means:

- restating the broad structure
- defending the middle carrier
- or reopening already-settled shell analysis

It means only:

- shrinking the remaining core
- killing it
- or cashing it into the asked medium

If one tiny exhaustive probe or small brute-force check appears near this point,
it counts only as one exact check or hostile witness on the current object.
It does not count as structural completion by itself unless it compresses back
into one smaller executable object, quotient, merge law, or exact readout body.

If that remaining core is still split between:

- one structural/object-side completion bite
- and one answer-side readout bite

do not force a fake choice in favor of readout merely because it looks nearer release.

While the same carrier still lacks the structural object it needs,
one local structural bite may hold authority ahead of the readout bite.

This is still late-stage closure, not route regrowth.

If a closure gate is already active but the current layer is still under
live `layerwise_reselection_pressure`, that closure gate may keep promotion
authority without yet becoming the winning executable move.
In that case the runtime should prefer one thinner structural/object-side
discharge before allowing the asked-medium readout to take execution authority.
If one thin interlayer bridge keeps closure authority live during that
discharge, the asked-medium seal should stay foregrounded rather than cooling
back into ordinary continuation.

If the runtime can already surface one active `discipline_contract`,
read it only as one local anti-regrowth veto:

- one already-bound bite should keep foreground authority
- one already-explicit handoff should keep reselection authority
- one already-concrete same-carrier bite should cool broad route reconstruction

This still does not authorize a workflow.
It authorizes only one same-layer refusal of ordinary regrowth
while one local bite is already concrete.

That veto must stay weaker than explicit local structure.
It must not override:

- one explicit primitive competition
- one explicit carrier handoff
- one explicit separating check
- or one explicit post-touch refusal

It may only stop decorative broadening around the current bite.
It may not choose the next layer, next winner, or next stage.

## Same-Carrier Persistence

Once one object has action authority,
the default should be to stay on that same carrier until one of three things happens:

1. one same-carrier change lands
2. one hostile witness kills the line
3. one exact check tears the object and hands authority to a smaller residue

Without one of those events,
switching into nearby explanation, rival narration, or prettier family language is drift.

If the run is resuming from a known late-stage sticking point,
the default should be even stricter:

- do not regrow the old shell
- do not restart from broad exploration
- do not spend long on why the middle carrier is promising

Resume directly from the smallest still-live residue unless one witness proves that residue was misidentified.

When stuck, briefly rebind:

- the ask
- the strongest still-true current object
- and the current explicit gap

Then either spend the next same-carrier bite,
or reseat authority only if that same gap survives more honestly on a thinner carrier.

## Asked-Medium Priority

When the object is already close enough that code, answer, proof text, file artifact,
or executable verification can be touched honestly,
that contact should be treated as part of the same object,
not as afterwork.

Near release, asked-medium contact outranks:

- one more family description
- one more closure-shaped summary
- one more elegant restatement of the gap

## Unsupported Is Still Open

The following are not near-passes.

They are still open:

- `UNSUPPORTED`
- placeholder branches
- "works only for small cases"
- "small brute force suggests the pattern"
- "the final step should be X"
- "only one last necessity / quotient / merge / readout remains"

If the asked medium still contains one of those forms,
the release veto stays active.

## Single-Gap Rule

Near the end, if the remaining burden can already be named as one explicit gap,
that gap should hold terminal authority alone.

Examples:

- one final necessity exclusion
- one exact state compression
- one exact merge law
- one exact readout formula
- one exact coexistence proof

Once the gap is this small, the run should no longer reward:

- broader structural summaries
- additional atmosphere about why the task is hard
- or new middle carriers that do not directly cut that same gap

## Gap Must Bind A Program

Once one late-stage gap is explicit,
that gap should not remain only as a named absence.

It should quickly bind to one tiny executable program on the same carrier or asked medium:

- write one exact answer-side object
- write one exact constructive pullback
- write one exact code-side branch or state
- check one exact dependency
- read out one exact formula
- or kill the gap by one hostile witness

If the gap still exists only as diagnosis,
terminal authority is still underbound.

Near release, one gap without one bound verb-object program
does not yet count as real action authority.

## Constructive Rival Exception

Single-slot preference is still the default.

But if the surviving dispute is no longer about shell language
and is now about realizability, coexistence, constructive pullback, or final-answer family,
one nearest constructive rival may stay warm beside the winning slot.

That rival does not count as free plurality.

It must bind its own:

- object or residue
- debt
- next bite
- and separating witness, coexistence carrier, constructive pullback, or exact readout object

If it cannot bind those,
it falls back to commentary pressure and should not consume late-stage authority.

If it can bind those,
it may persist only until:

- the winning slot kills it
- it kills the winning slot
- or the family split collapses into one smaller executable residue

## Bound Commitment Is Working Memory

Late-stage working memory should stay stratified.

Keep:

- one bound commitment object in foreground
- at most one warm constructive rival beside it
- and all other lines only as cooled pressure or archived residue

If several lines all still feel equally foregrounded,
the working slot is not yet honestly bound.

## Commitment Must Carry One Revocation Handle

A bound slot is still incomplete if it cannot say
what exact event would tear it and return authority to a smaller residue.

That revocation handle should usually be one of:

- one hostile witness
- one exact check
- one asked-medium failure
- or one coexistence / certification failure

Without one revocation handle,
the slot is too comfortable to deserve late-stage confidence.

## History-Shaped Carrier Is Still Open

A carrier is still not executable if its control still depends on:

- an ever-growing stack
- the whole past chain
- one unquotiented history object
- one barrier history
- one remembered sequence of local pivots
- or one state family that has not yet collapsed to a finite executable quotient

Such a carrier may be a very strong middle object.

It is still open unless the run does one of:

- compress it to a finite state / quotient
- extract one exact merge law from it
- extract one exact readout from it
- or kill it with one witness showing that no safe quotient has yet been found

Near release, "I now understand the stack / chain / barrier picture"
does not count as closure.

## Final Rule

This layer should stay tiny.

If action authority needs many visible rules,
the host is still compensating with packaging instead of absorbing control.
