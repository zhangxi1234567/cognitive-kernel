# Runtime Overview

The runtime should consume the constitution as a background field, not as a checklist.

Use the adjacent runtime files as thin specialist correctives, not as a menu:

- `FALSE_ESSENCE_ELIMINATION.md` for killing impostor essences
- `CONFLICT_RESOLUTION.md` when multiple candidates remain alive
- `ACTION_AUTHORITY.md` only when the live controller is already visible but execution authority is dissolving before the next object-side move or asked-medium contact
- `CONTROL_STATE.md` when several runtime surfaces are starting to sound aligned but may no longer share one small live state face
- `WORKING_MEMORY_CONTROL.md` when foreground binding, slot crowding, or writable-surface loss is the issue
- `CALIBRATION_AND_REFLUX.md` when the issue is whether the next spend should separate, materialize, or shrink locally after output/check failure
- `MEMORY_GUIDED_REUSE.md` when prior runs may lawfully leave thin competition texture without becoming route replay

One optional executable guard may support these thin surfaces without turning them into workflow:

- `tools/runtime_guard.py` for validating release-veto conditions against one small state file
- `tools/runtime_state.py` for creating and mutating that same small state without prescribing a route
- `tools/runtime_state.py bind-local` for explicitly projecting one already-local monitor state into one tiny local bite
- `tools/runtime_state.py rebind-local` for explicitly projecting one already-local thinner-carrier primitive reselection into one tiny handoff
- `tools/runtime_state.py spend-local` for explicitly collapsing one already-live thinner-carrier handoff into one next primitive bite on that new carrier
- when a live skill-authority bridge already compresses to one executable local touch, bind/spend may consume that one touch too, but only as a one-bite authority bridge
- if skill-side authority and primitive-side touch disagree on the same local bind/spend, the executable layer should refuse rather than silently choosing a route
- those one-shot local bridges should also refresh the tiny primitive field to the current live layer whenever authority moves
- stale primitive competition from an older layer should cool out instead of blocking the new layer's primitive spend
- `tools/runtime_next_touch.py` for one-shot consumption of the default local action readout when that bridge exists
- `tools/runtime_reselection.py` for one-shot readout of a local carrier-change / primitive-reselection handoff when that bridge exists
- `tools/runtime_self_check.py` for one-shot readout of the current self-check focus when the state is valid
- `tools/runtime_primitive.py` for one-shot readout of the current primitive field / competition surface
- `tools/runtime_controller.py` for one-shot readout of the current top-level control signal surface
- `tools/runtime_inhibition.py` for one-shot readout of the current local inhibition owner / promoted move surface
- `tools/runtime_catalog.py` for a direct readout of the currently executable primitive/skill families and their alias surface
- `runtime/PRIMITIVE_SEMANTICS.md` and `runtime/primitive_semantics.json` for the thin mechanism-facing semantics of live primitive families

`tools/runtime_consume.py` should stay inspect-first.
If it ever persists one local bind, that should require an explicit one-shot opt-in rather than being the default read path.
If `--spend-handoff` is requested and the spend is not concrete enough, that refusal should surface as a refusal rather than collapsing back into a normal inspect result.
If one active `discipline_contract` is already asking for a real local bind or spend,
an inspect-only success should not be confused with runtime consumption evidence.
The smallest honest evidence is one real one-shot runtime transition,
not a descriptive readout alone.

That small executable layer may expose one tiny `control_bridge`
containing one `next_touch`.
While `release_veto` is active, that bridge may mark `next_touch`
as the default local action.
That should remain true only for an explicit current-carrier bind.
If the guard is merely exposing a derived local candidate, that candidate should stay descriptive rather than becoming the default host action.
If a monitoring pressure has already purchased that local action,
the same bridge may also carry one tiny gate-binding annotation saying
what focus bought authority and what nearby continuation was demoted.
If the state is already local enough but no explicit `bound_program` has been written yet,
the guard may still derive one temporary `next_touch` readout from the same state face.
That does not authorize chaining or route growth;
it only prevents monitoring from stopping at commentary when one bite is already obvious.
After release, that bridge should disappear again unless a new vetoed bite is bound.
It should still stay beneath workflow scale.
Released state should not keep active handoff / gate / primitive-control artifacts alive.

The same executable layer may also expose one tiny primitive field.
That field should say only:

- what the current layer object is
- which one or two primitive families are still honestly live there
- and what one cheap tie-break surface could still revoke fake progress early
- and, when available, whether those candidates came from explicit local hints, agenda-side hints, or only text fallback

It exists only so "reselect primitives on the thinner carrier" becomes locally readable.
It must not become a tactic shelf or an explicit solve order.
If more than one primitive is still live and no tie-break surface exists yet,
that field should stay observational rather than collapsing early into one next primitive touch.
If explicit primitive hints already exist on the current thinner layer, those hints should outrank text-only guessing.
When host or guard readouts surface primitive semantics, those semantics should stay descriptive:
mechanism, controller-question, wake signals, cheap honest touch, and misuse warning.
They must not become a route order.

The same executable layer may also expose one tiny local `skill` capability surface.

That surface should say only:

- which skill families are currently live on this layer
- which skill families are competing locally if more than one is still honest
- which skill families are currently demoted by local inhibition
- and which skill family, if any, currently holds closure authority

It exists only so skill-level wake-up can become locally readable without turning into
a skill menu or route scheduler.
It must not encode:

- skill order
- task-type-to-skill lookup
- multi-step skill chaining
- or a solve-time dispatcher tree

If one local skill family already has honest closure authority on the current layer,
the runtime may also expose one tiny `skill_authority_bridge`.

That bridge should say only:

- which skill currently owns one local executable touch
- what that one executable local touch is
- and whether that skill should cool after the touch lands

It must not schedule what happens after that touch.

One current layer may still keep a tiny local skill combination warm around that
single authorized touch.

That surrounding combination may say only:

- which one or two structural / readout / witness / closure families are still active together on this layer
- which of them currently owns the executable touch
- which of them remain only as supporting background or local rivalry

It exists only so a layer can stay compressive and fast without pretending that
all nearby skill support vanished the moment one bite became executable.
It must not become a skill script, stage order, or multi-step executor.

If one live `discipline_contract` is also present, read it only as a brake on
broad ordinary regrowth around that already-concrete bite.
It must not decide:

- which primitive or skill wins
- whether a thinner carrier should take over
- what should happen after the current bite lands

Explicit local competition, explicit handoff, and explicit separating checks
must keep their own authority.
`discipline_contract` may cool drift around them; it must not replace them.
When that contract is active on the current layer, broad explanatory regrowth
should be treated as noise rather than neutral continuation.
Host-facing inspect surfaces may cool wider route-reconstruction readouts until
that bite is bound, spent, or honestly torn.
That quieting should stay consistent across controller, primitive, and
reselection readouts on the same current layer.
Once one local owner is active, sibling readers should cool descriptive surplus
rather than re-amplifying stale competition, stale shell language, or older-layer pressure.

If `layerwise_reselection_pressure` is still active and direct closure is not yet allowed,
`exact_closure` may remain promoted as a live control pressure without becoming
the current winning executor.
In that case, closure can stay warm while one thinner structural discharge
still holds executable authority first.

Any tiny exhaustive probe, small-case enumeration, or pattern-mining move
should count only as a microscope on one already-live primitive or skill
hypothesis on the current layer.
It may separate, kill, or sharpen that local hypothesis.
It must not become the main route, a fallback solver, or a second broad line
that steals ownership from explicit competition, handoff, or closure authority.

If a thinner-carrier structural bite is still honestly unpaid while
`exact_closure` is already pulling on the asked medium,
the runtime may also expose one tiny `interlayer_discharge_bridge`.

That bridge should say only:

- that one structural bite may be spent first
- what that one spend-first program is
- what closure-side touch should remain live afterward
- and whether closure authority should stay foregrounded

It exists only to let one thinner-carrier debt be paid before closure seals.
It must not become a stage machine, replay queue, or future-step scheduler.
If closure authority stays foregrounded there, the structural spend may happen
first while asked-medium closure remains the next honest seal rather than
cooling back into ordinary continuation.

The same executable layer may also expose one tiny derived `control_signals` readout.
That readout should say only:

- whether the current run still looks surface-heavy or controller-bound
- whether ownership is still unlocked, only narrowing, or already locally bound
- whether the current carrier still looks too thick, is thinning, or is thin enough
- whether the current object is under shell suspicion or beautiful-middle risk
- whether thinner-carrier primitive reselection is now locally due
- and which one or two primitive families deserve extra local wake-up pressure
- and whether counter-question / closure-gate / supervisory-pulse / god-view / metacognitive / central / hindbrain-style control is currently live
- and what local attraction / reward / penalty field is currently pulling, promoting, or demoting behavior
- and, when useful, one tiny current-layer `micro_control_surface` for closure / reward / penalty / supervision / counter-question
- and, when a primitive field or primitive competition is already live, what the local primitive-control face currently is before heuristic pressure is added
- and, when tiny probing is live, one `probe_discipline` that keeps that probe bound to a live skill hypothesis
- and, when that discipline is active, tiny probing should validate the live skill line rather than becoming the main route

It exists only to bias primitive competition back toward essence / owner / thinner-carrier honesty.
It must not become a planner, route script, or explicit multi-step tactic policy.
If explicit current-layer primitive state already exists, the control readout should mirror that state first and only let heuristic pressure fill remaining gaps.
One current-layer `layerwise_reselection_pressure` may also surface there.
That object should say only whether the current layer still looks too thick,
too beautiful-middle, or too weakly grounded to settle directly.
It may wake a few current-layer primitive or skill families again,
but it must not encode a fixed solve order.

When a default local action is already consumable,
the host-facing consume surface may also carry the same thin:

- `meta_controls`
- `incentive_field`
- `micro_control_surface`
- `closure_nucleus`
- and `inhibition_state`

so the action readout does not drift away from the control-side brain-region surface.

That same thin consume / control / primitive surface may also expose one tiny derived
`closure_nucleus` when late closure is already the real owner.

That nucleus should say only:

- one current object
- one current unpaid debt
- one asked-medium surface
- one revocation handle
- one current structural/object-side bite if any
- one current readout bite if any
- one separating check if any
- and whether same-carrier closure still forbids route growth

It exists only to keep several readouts touching the same local closure body.
It must not become a staged executor or an explicit route chooser.

If the current unpaid debt itself has become nameable as one smaller honest object,
the same executable layer may also expose one tiny `gap_object`.

That readout should say only:

- what smaller object the current missing thing has collapsed into
- what debt/source phrase it came from
- one cheap same-carrier check or compression touch
- and one or two primitive hints that now deserve local wake-up

It exists only so "missing thing" can inherit local authority without reopening a broad shell.
It must not become a phase label, a route planner, or an automatic retry ladder.

If the run is locally stuck, that same layer may also expose one tiny `resume_bridge`.

That readout should say only:

- the current ask surface
- the strongest still-true known object
- the current explicit gap
- whether the cheapest honest continuation is same-carrier continuation or thinner-carrier reselection
- and which tiny supporting skill combination is still locally relevant

It exists only so a stuck run can rebind ask + object + gap before spending again.
It must not become a phase machine, a restart protocol, or a route tree.

That `micro_control_surface` must stay thin:

- it may describe one current closure debt
- one current reward pull
- one current penalty demotion
- one current supervisory owner
- and one current counter-question

It must not describe:

- a phase order
- a route winner
- a retry ladder
- or a future solve sequence

If the local primitive split itself is already concrete,
that same executable layer may also carry one tiny primitive competition object.
That object should say only:

- what the current layer object is
- which one or two candidates are still live there
- what one cheap separating check could split them
- and, if already settled, which candidate has won

It exists to make primitive rivalry locally falsifiable.
It must not become a search tree, tactic shelf, or route tournament bracket.

If an explicit primitive competition is still live on the current layer,
that current-layer competition should also outrank a merely derived reselection bridge.

The runtime may still bind one structural/object-side local bite from that same-layer competition
before falling through to a thinner-carrier handoff,
as long as it stays one bite, one carrier, one debt, and one revocation handle.

If the run has already crossed into a thinner carrier,
that small executable layer may also expose one tiny reselection bridge.
If the state is already local enough but no explicit handoff has been written yet,
the guard may still derive one temporary reselection readout from the same state face.
That bridge should say only:

- what local trigger moved authority
- what pressure won locally
- what pressure cooled if any
- and what still-live tiny warm field remains nearby

It must not choose a route family or schedule future moves.

The same executable layer may also expose one tiny self-check agenda readout.
That readout should surface only one current focus:

- seam
- rival
- carrier
- or asked-medium

It must not become a checklist or a staged solver shell.

If one thinner-carrier handoff is already explicit or locally derivable,
that self-check agenda should cool rather than compete with the handoff.
At that point the single live readout should be the reselection bridge,
not a second advisory lane.
Likewise, `bind-local` should not silently choose between a same-carrier bind
and a thinner-carrier handoff when both are concrete; that split must stay explicit.

The same executable layer may also expose one tiny inhibition readout while `release_veto` is active.
That readout should say only:

- which local owner currently has temporary gate authority
- which exact local move is promoted
- which non-contact continuations are demoted
- and what event tears that temporary gate

It must not schedule retries, accumulate stages, or become a planner.

It is acceptable to treat this as a parameter-like cognition layer:

- a reusable shaping of what gets noticed
- a reusable shaping of what gets distrusted
- a reusable shaping of what gets minimally but sufficiently verified
- a reusable shaping of when to stop widening the fix
- a reusable shaping of when not to mistake a local pattern for the mother law
- a reusable shaping of when one abstract boundary requires sibling completion
- a reusable shaping of when not to lift the explanation above the best evidence layer
- a reusable shaping of essence as the lowest evidenced complete boundary
- a reusable shaping of who had final authority to preserve or reject attached metadata on the result
- a reusable shaping of the difference between producing a result and validating what metadata may lawfully survive on it
- a reusable shaping of the difference between reattaching metadata and proving that the metadata is still valid for this exact result

It is not acceptable to treat it as a pipeline that mass-produces answers by named trick recall.

It is also not acceptable to treat a short-lived late-stage rival lane
as permission to regrow a visible multi-lane workflow.

Local rival materialization is allowed only when:

- the same live burden still splits across real final-answer families,
  certification families, readout families, or realizability families
- one surviving rival still deletes a different part of that same burden
- and keeping that rival alive makes the remaining seam smaller rather than larger

If those conditions are absent,
plurality should cool back down rather than institutionalize itself.

## What It Is

- a compact set of cognitive laws
- domain-agnostic
- aimed at attention, not procedure
- compatible with math, coding, research, writing, and diagnosis

## What It Is Not

- not a fixed order of moves
- not a library of named methods
- not a requirement to narrate internal reasoning
- not a trigger map from task type to tactic

## Consumption Rule

The runtime may use these laws to:

- notice tension
- distrust decorative shell
- prefer smaller truthful carriers
- test claims at their weakest seam
- stop when further reduction would turn dishonest
- collapse back into the exact asked interface before claiming completion
- prefer the narrowest repair surface that already explains and operationally closes the observed failure
- strip label-level or family-level costumes before accepting a proposed essence
- when the essence is a shared boundary invariant, do not stop until sibling entry points on that same boundary are either closed or honestly ruled out by witness
- stay at the lowest honest evidenced boundary unless it fails to explain or close
- seek the first layer where explanation and completion coincide, not merely the lowest or highest layer alone
- do not let "minimal" mean "finished" while the same evidenced law remains open nearby on the same boundary
- do not let "deeper" outrank "more correct for this witness"; upstream drift is not a win by itself
- when multiple layers touch the same carried label or metadata, prefer the layer that finally authorizes whether it may stay attached to the result
- when one layer converts the object and another validates the carried metadata, treat the validator as the default owner of the bug
- when one layer carries metadata forward and another owns the exact legality test for keeping it, treat the legality-test owner as the default owner of the bug
- when a candidate feels elegant but unearned, use `FALSE_ESSENCE_ELIMINATION.md` to try to kill it before it settles
- when multiple candidates remain alive after first pressure, use `CONFLICT_RESOLUTION.md` to separate them by witness and closure rather than taste

The runtime must not turn them into:

- numbered stages
- mandatory prompts
- visible ritual
- exhaustive multi-law sweeps
- a habit of repairing the whole neighborhood after one local controller has already been evidenced
- a habit of canonizing the first elegant local pattern as the final explanation
- a ritual of enumerating every sibling path even when no shared boundary law has been evidenced
- a habit of replacing a directly evidenced low-level bug with a cleaner but less evidenced higher-level story
- a habit of mistaking the smallest local patch point for essence when the same invariant is still incomplete nearby
- a habit of walking past the first sufficient boundary into a broader upstream story that does not improve present-task closure
- a habit of confusing the place where a label is inferred with the place where that label is finally allowed to survive on the result
- a habit of confusing the layer that produced a result with the layer that judged whether old metadata could still remain true on that result
- a habit of stopping at the layer that reattached metadata instead of the layer that actually decides whether the reattached metadata is still lawful

If the constitution becomes easier to command than to absorb, it has already regressed into workflow.
